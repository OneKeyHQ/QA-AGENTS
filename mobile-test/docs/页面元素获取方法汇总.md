# 页面元素获取方法汇总

本文档汇总当前项目中「如何拿到页面上的元素并用于 E2E」的几种方式，以及在 Page 里怎么写选择器。

---

## 一、获取元素信息的途径（怎么知道页面上有什么）

### 1. App 侧加 testID（优先推荐）

| 做法 | 说明 |
|------|------|
| **谁做** | App 开发在可交互控件上加 `testID`（RN）或各平台等价属性 |
| **结果** | Android 会变成 `resource-id`，iOS 变成 `name`，E2E 用 `api.by.id('xxx')` 即可 |
| **优点** | UI 改布局也不易失效，不依赖抓 DOM |
| **规范** | 命名建议：`页面/模块_用途`，如 `onboarding_create_wallet_btn` |

---

### 2. Appium Inspector 手动抓 DOM

| 做法 | 说明 |
|------|------|
| **步骤** | 用 Appium Inspector 连上设备/模拟器 → 打开目标页面 → 看层级/XML，抄 `resource-id`、`text`、`content-desc`、bounds 等 |
| **写哪里** | 在 `pages/xxxPage.js` 里手写 getter，用 `api.by.id()` / `api.by.text()` / `api.by.xpath()` |
| **缺点** | 每个页面都要手动做一次，UI 一改又要重新抓 |

---

### 3. 录制 + 轮询（半自动）

| 做法 | 说明 |
|------|------|
| **命令** | `yarn test:android:record`（静默）或 `yarn test:android:record:debug`（看诊断） |
| **流程** | 跑用例 → 在设备上**手动点击**要录的元素（建议点会跳转/弹窗的按钮）→ 轮询对比 `getPageSource()` 变化，推断被点击元素 → **等用例跑完**（例如 60 秒结束或手动结束），结束时生成/追加到页面文件并可用编辑器打开定位到新增行 |
| **写入路径** | 若已有 `pages/<pageName>/` 子目录（如 `pages/onboarding/`），则写入 **`pages/<pageName>/<pageName>Page.js`**；否则写入 **`pages/<pageName>Page.js`**。例如 `pageName: 'onboarding'` → `pages/onboarding/onboardingPage.js`。**必须等测试结束**（`stopRecording()` 被调用）才会写文件，中途退出不会生成文件。 |
| **配置** | 在 `test/recordManualClicks.e2e.js` 里改 `pageName`、`api.pause(毫秒)`；可选 `openEditorAtLine: true`、`RECORD_DEBUG=1` 排查 |
| **适用** | 新页面或改版后快速补一批 getter/方法，再人工整理命名和去重 |

详见：[元素定位策略 - 2.2.1 录制+轮询操作步骤](./ELEMENT_LOCATOR_STRATEGY.md#221-如何用录制--轮询走一遍新流程生成或更新-page)、[RECORDING_USAGE.md](../RECORDING_USAGE.md)。

---

### 4. 按需 dump 当前页 XML（脚本化）

| 做法 | 说明 |
|------|------|
| **思路** | 在测试或脚本里，进入目标页面后调 `driver.getPageSource()` 把当前页 XML 存下来（可按页面名/时间命名） |
| **后续** | 用解析脚本从 XML 里提 `resource-id`/`text`/`content-desc` 等，按 POM 规则生成候选 getter/选择器，再人工合并进 Page |
| **适用** | 批量抓多页、或和录制互补（例如只 dump 不点击） |

---

### 5. 从已有 XML/常量提取

| 做法 | 说明 |
|------|------|
| **来源** | 如 `const/xmlPageElements.js`、`xmls/*.xml` 等从某次 dump 或导出得到的结构 |
| **用法** | 作为「候选元素定义」参考，手写到 Page 或通过脚本转成 getter/选择器再合并 |
| **注意** | 若 UI 已改版，需重新 dump 或跑录制更新 |

---

## 二、在 Page 里怎么写选择器（拿到信息之后）

元素信息有了之后，在 `pages/xxxPage.js` 里用下面 API 写成 getter，供用例调用。

### 2.1 定位 API（`api.by.xxx`）

| API | 用途 | 示例 |
|-----|------|------|
| **api.by.id(str)** | 用 resource-id（Android）/ name（iOS）/ data-testid（Web） | `api.by.id('nav-header-close')` |
| **api.by.idStartWith(str)** | resource-id/name 前缀匹配 | `api.by.idStartWith('select-item-')` |
| **api.by.idsStartWith(str)** | 同上，返回多个元素 | `api.by.idsStartWith('select-item-')` |
| **api.by.text(texts)** | 按文本匹配（支持多语言，中文优先） | `api.by.text(['添加钱包', 'Add wallet'])` |
| **api.by.xpath(str)** | 任意 XPath | `api.by.xpath('//android.widget.Button[@text="确定"]')` |

多语言时优先用 `api.by.text([中文, English])`，框架会先试中文再试英文。

### 2.2 选择器优先级（POM 约定）

从高到低建议：

1. **text / content-desc**（最稳，且易做中英文）
2. **resource-id**（有 testID 时用 `api.by.id()`）
3. **相对 XPath**（无上述时再用，易受结构变化影响）
4. 避免：纯 bounds、绝对坐标

详见：[POM_RULES.md](../POM_RULES.md) 定位优先级与多语言示例。

## 三、对照小结

| 目的 | 推荐方式 |
|------|----------|
| 长期稳定、少维护 | App 侧统一 testID，E2E 用 `api.by.id()` |
| 新页/改版快速出稿 | 录制 + 轮询（`yarn test:android:record`）→ 再人工整理 |
| 单页或临时查元素 | Appium Inspector 抓 DOM，手写 getter |
| 批量抓多页 | 脚本里 getPageSource() 存 XML → 解析生成候选 → 合并进 Page |
| 无 ID 的图标/控件 | 仅作兜底时考略 Appium 的 -image 图像定位（见 ELEMENT_LOCATOR_STRATEGY 文档） |
