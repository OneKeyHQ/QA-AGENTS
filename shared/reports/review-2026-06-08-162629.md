### 阻断问题（block）

#### BLOCK-1：脚本覆盖映射引用了不存在的用例文档

- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:2`
- 当前内容：`// covers: docs/qa/testcases/cases/transfer/2026-05-20_Transfer-Kaspa转账-软件钱包.md §1 参数表`
- 问题：仓库中不存在 `2026-05-20_Transfer-Kaspa转账-软件钱包.md`，实际存在的是 `docs/qa/testcases/cases/transfer/2026-05-20_Transfer-Kaspa转账-HD钱包.md`。
- 风险：Dashboard/审查无法追溯脚本与用例来源，后续同步规则会误判脚本与文档脱节。
- 建议：修正头部 covers 路径，并确认 Extension 脚本是否确实覆盖 HD 钱包文档，或补充专门的 Extension/软件钱包用例文档。

#### BLOCK-2：发送成功后的历史记录字段校验是软断言，字段缺失也会通过

- 文件：`src/tests/helpers/transfer.mjs:227-239`
- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:877-879,903-905,929-931,955-957`
- 问题：`verifyHistoryRecord()` 只把命中的字段 push 到 `fields` 后直接返回；调用方只输出 `fields=${fields.join(',')}`，没有断言必须包含金额、类型、状态、时间、from/to、哈希、网络费用、网络等用例要求字段。若 `fields=[]` 或只命中 `token`，步骤仍会标记 passed。
- 对照用例：`2026-05-20_Transfer-Kaspa转账-HD钱包.md:32` 要求历史记录显示金额、类型、状态、时间、from/to、哈希、网络费用、网络。
- 建议：`verifyHistoryRecord()` 接收 expected 字段集，缺任一必填字段时 throw；Kaspa 可按链特性允许代币关联记录多条，但不能放宽到“有任意记录即可”。

#### BLOCK-3：EXT-KASPA-001 未断言“预览按钮不可点”

- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:664-669`
- 问题：用例要求金额 0 时“显示最小金额 0.2 提示 + 预览按钮不可点”，脚本只检查页面文本包含 `0.2` 或 `最小`，没有验证 `page-footer-confirm` 不存在、禁用、不可点击或文案不允许预览。
- 风险：如果错误提示显示但按钮仍可提交，脚本仍会通过，漏掉资金安全关键拦截缺陷。
- 建议：在同一步补充按钮状态断言：确认预览按钮不存在或 `aria-disabled=true` / disabled / pointer-events none / 文案为不可提交状态。

#### BLOCK-4：成功转账流程未覆盖用例要求的关键确认页与区块浏览器断言

- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:899-906,925-932,951-958`
- 对照用例：`docs/qa/testcases/cases/transfer/2026-05-20_Transfer-Kaspa转账-HD钱包.md:32-36`
- 问题：成功发送用例只执行预览、确认、历史字段摘要；缺少明确断言：法币切换展示、确认页网络/发送地址/金额与 icon/收款地址、网络费用展示更新、最大值等于最大可转且扣 Gas 预留、区块浏览器金额/symbol/费用/from/to 对账。
- 风险：脚本通过不代表文档预期通过，属于 1:1 用例对照缺失。
- 建议：按文档预期拆成独立 `safeStep`，每个字段读取失败时 throw；无法自动化的区块浏览器跳转需在步骤里明确 skipped 并写原因，不应静默省略。

### 警告问题（warn）

#### WARN-1：`switchNetwork()` 仍使用 `fill()` 输入 React 搜索框

- 文件：`src/tests/helpers/network.mjs:24`
- 问题：项目规则要求 OneKey React 输入框优先使用 `locator.pressSequentially()`，避免 `fill()` 不触发搜索/过滤。
- 建议：清空后使用 `pressSequentially(networkName, { delay: 30~80 })`，并轮询网络结果。

#### WARN-2：脚本头部仍声明 `KASPA_TEST_RECIPIENT` 覆盖，但实现已移除读取逻辑

- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:3`
- 问题：diff 移除了 `resolveRecipientAddress()` 和 `process.env.KASPA_TEST_RECIPIENT` 读取，实际收款地址来自 runtime-config 账户复制，但注释仍说可用 `KASPA_TEST_RECIPIENT` 覆盖。
- 建议：更新注释，避免 QA 误以为环境变量仍生效。

#### WARN-3：EXT-KASPA-002 存在专用大段逻辑，建议抽成通用费用档 helper

- 文件：`src/tests/extension/transfer/kaspa/transfer.test.mjs:700-875`
- 问题：费用档切换逻辑较长，包含预览点击、费用弹窗等待、SegmentControl 坐标点击、保存确认等通用行为。后续其他链或同脚本复用时易产生分叉维护。
- 建议：提取为 `switchFeeLevelsAndSubmit(page, token, levels)` 或放入 transfer helper，保留 Kaspa 特例参数。

#### WARN-4：`verifyHistoryRecord()` 兜底点击首条交易，可能误验非本次交易

- 文件：`src/tests/helpers/transfer.mjs:207`
- 问题：如果未找到包含目标 token 的记录，会点击 `rows[0]`。刚完成发送后历史异步刷新或代币关联记录多条时，首条可能不是本次交易。
- 建议：至少要求首条详情包含 token/金额/状态中的核心字段；更好是传入发送前后的时间或金额，用更多信号锁定目标记录。
