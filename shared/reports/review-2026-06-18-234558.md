### 阻断问题（block）

1. `src/tests/extension/transfer/ada/transfer.test.mjs:938` / `src/tests/extension/transfer/ada/transfer.test.mjs:948`，`src/tests/extension/transfer/kaspa/transfer.test.mjs:870` / `src/tests/extension/transfer/kaspa/transfer.test.mjs:1054`，`src/tests/extension/transfer/kaspa/transfer.test.mjs:1089` / `src/tests/extension/transfer/kaspa/transfer.test.mjs:1095`，`src/tests/extension/transfer/kaspa/transfer.test.mjs:1161` / `src/tests/extension/transfer/kaspa/transfer.test.mjs:1165`，`src/tests/extension/transfer/kaspa/transfer.test.mjs:1204` / `src/tests/extension/transfer/kaspa/transfer.test.mjs:1210`
   - 问题：最终确认提交失败后，用例仍会继续进入历史记录校验。
   - 依据：`safeStep()` 在失败时只记录 failed 并返回 `false`，不会抛出终止用例；但这些 `预览并确认` / `预览页切换费用档并确认` 步骤没有接收返回值，后面直接执行 `verifyHistoryRecord()`。
   - 影响：如果本次新增的 toast 失败识别正确抛出 `提交后出现失败提示: ...`，用例仍会继续等历史记录，最终结果可能同时出现“提交失败提示”和“未匹配历史记录/缺少 historyAmount”等二次错误，正好会把用户想要回传的真实失败原因冲淡。
   - 建议：所有依赖提交成功的历史校验前都加门禁，例如 `const submittedOk = await safeStep(...); if (!submittedOk) return;`；同时要求 `submitted` / `submittedAmount` 必须存在才进入历史校验。Kaspa `EXT-KASPA-002/003/005/006` 和 ADA `EXT-ADA-002/003/005/006` 都需要覆盖。

### 警告问题（warn）

1. `src/tests/extension/transfer/ada/transfer.test.mjs:39` 与 `docs/qa/testcases/cases/transfer/2026-05-20_Transfer-ADA转账-HD钱包.md:15`
   - 问题：ADA 自动化参数已扩展为 6 条，包括 MELD/ADA `0.9` 最低转账金额拦截，以及 BANK `Max`；但源用例文档仍是 4 条，且 BANK 仍写金额 `1`。
   - 影响：脚本行为来自本轮用户要求，运行层面可以接受；但 `covers:` 指向的文档没有同步，后续 review / 回归清单会看不出新增拦截覆盖来源。
   - 建议：补充 ADA 用例文档参数表与主流程行，或在脚本头部明确标注这两条为临时自动化增强项及来源。

2. `src/tests/helpers/extension-transfer.mjs:515`
   - 问题：toast 失败判断依赖轮询中观察到 `seenBusyConfirm`，但最终确认点击后第一个采样发生在 helper 循环内；如果按钮 loading 很短，可能已经恢复可点并出现新 toast，此时 `seenBusyConfirm=false`，不会立即判定失败。
   - 影响：少数快速失败场景可能退化成反复重试或最终超时，无法稳定把 toast 文案反馈到提交步骤。
   - 建议：保留“必须有提交尝试/响应”的原则，但在点击最终确认后立即采样一次按钮状态，或把 `clickFinalAction` / 初始 final click 成功记录为 `submitAttempted`，再结合新 toast、预览仍打开、确认按钮恢复可点做失败判断。

3. `src/tests/helpers/no-eval-cdp.mjs:61` 与 `src/dashboard/test-executor.ts:202`, `src/dashboard/test-executor.ts:288`, `src/dashboard/test-executor.ts:335`
   - 问题：Dashboard 会在连接后、setup 前、每个 case 前重复安装 no-eval fallback，这是必要的；但 `installNoEvalSafeEvaluate()` 每次都会创建新的 CDP session，`PATCH_KEY` 只做标记，没有复用 session。
   - 影响：长队列连续执行插件端用例时会累积 CDP session，增加资源占用和偶发连接不稳定风险。
   - 建议：把 CDP session 也挂在 page 的 symbol 上复用，同时每次仍重新覆盖 `page.evaluate/page.waitForFunction`，满足 K-162 的“每个 case 前重装”要求。

4. `src/tests/helpers/no-eval-cdp.mjs:63`
   - 问题：兼容层把 `page.waitForFunction()` 的返回值改成了普通值，而 Playwright 原生返回 `JSHandle`。
   - 影响：当前扫描到的扩展转账路径只 `await page.waitForFunction()`，未使用返回值；但这个 fallback 由 Dashboard 对所有插件页生效，后续若有插件端脚本依赖 `.jsonValue()` 会被破坏。
   - 建议：在 helper 注释中明确该兼容层的 API 差异，或返回一个最小兼容对象；至少在插件端脚本规范里避免依赖 waitForFunction 的 JSHandle 返回。
