# 冒烟测试与独立分析指令规范

> 本文档定义冒烟测试和独立分析（perf/ws/analyze）的执行规范。
> 
> **依赖**：执行时需配合 `chrome-mcp.md` 中的工具规范。

---

## 1）独立分析指令（性能 / WebSocket）

> 当你只想分析页面性能或 WebSocket 数据，而不需要执行完整的冒烟测试时，使用以下独立指令。

### 触发指令

| 指令 | 说明 | 适用场景 |
| --- | --- | --- |
| `perf <URL>` | 仅采集性能数据 | 快速检查页面加载性能 |
| `ws <URL>` | 仅监听 WebSocket 数据 | 调试实时数据推送（Perps/Market） |
| `analyze <URL>` | 性能 + WS 综合分析 | 全面诊断页面问题 |

### 指令对比

| 指令 | 功能验证 | 性能采集 | WS 监听 | 输出报告 |
| --- | --- | --- | --- | --- |
| `smoke <URL>` | ✅ | ✅ | ✅（仅图表数据相关） | 完整冒烟报告 |
| `perf <URL>` | ❌ | ✅ | ❌ | 性能分析报告 |
| `ws <URL>` | ❌ | ❌ | ✅ | WS 监听报告 |
| `analyze <URL>` | ❌ | ✅ | ✅ | 综合分析报告 |

---

### 1.1）perf 指令 - 性能分析

**用途**：仅采集页面性能数据，不执行任何功能操作。

**执行流程**：
1. 连接浏览器（Chrome DevTools MCP）
2. 导航到目标 URL（如果页面已加载，先刷新页面）
3. 在页面加载过程中注入性能采集脚本
4. 等待页面加载完成
5. 进入交互式监控模式
6. 持续采集性能指标（Core Web Vitals、加载性能、资源性能、网络性能）
7. 用户输入 `stop` 或 `结束` 停止监控
8. 输出性能分析报告

**交互式监控**：
- 执行 `perf <URL>` 后，进入监控模式
- 系统会自动刷新页面以确保能够采集 LCP（最大内容绘制）
- 用户可在页面上进行操作（点击、输入等），系统实时采集 FID（首次输入延迟）
- 用户可随时输入 `stop` 或 `结束` 停止监控
- 停止后输出完整报告

**LCP 和 FID 采集说明**：
- **LCP**：需要在页面首次加载时采集，系统会自动刷新页面以确保采集到 LCP 数据
- **FID**：需要用户实际交互（点击、输入等）才能触发，建议在监控模式下进行至少一次交互操作
- 如果 LCP/FID 未采集到，系统会在报告中标注原因并提供建议

**输出报告格式**：
```markdown
# <页面> - 性能分析报告

> 分析时间：YYYY-MM-DD HH:mm:ss
> 分析 URL：<URL>

## Core Web Vitals
| 指标 | 实测值 | 目标值 | 状态 |
| --- | --- | --- | --- |
| LCP | X.XXs | < 2.5s | ✅/❌ |
| FID | XXms | < 100ms | ✅/❌ |
| CLS | 0.XXX | < 0.1 | ✅/❌ |

## 加载性能
| 指标 | 实测值 | 说明 |
| --- | --- | --- |
| FCP | X.XXs | 首次内容绘制 |
| TTI | X.XXs | 可交互时间 |
| DOMContentLoaded | X.XXs | DOM 加载完成 |
| Load | X.XXs | 页面完全加载 |

## 资源分析
| 类型 | 数量 | 总大小 | 平均耗时 |
| --- | --- | --- | --- |
| JS | XX | X.X MB | XXXms |
| CSS | XX | X.X MB | XXXms |
| 图片 | XX | X.X MB | XXXms |

## 慢资源 Top 5（耗时 > 500ms）
| 资源 URL | 类型 | 大小 | 耗时 |
| --- | --- | --- | --- |
| /static/js/main.js | JS | 1.2MB | 850ms |

## 优化建议
1. [问题描述] → [优化建议]
```

### 1.2）ws 指令 - WebSocket 监听

**用途**：仅监听页面的 WebSocket 连接和消息，用于调试实时数据推送。

**适用模块**：Perps、Market、Swap（有实时数据的模块）

**执行流程**：
1. 连接浏览器（Chrome DevTools MCP）
2. 注入 WebSocket 监听脚本（页面加载前）
3. 导航到目标 URL
4. 等待 WebSocket 连接建立
5. 持续监听消息（默认 30 秒，可通过 `ws <URL> --duration 60` 指定）
6. 输出 WS 监听报告

**交互式监听**：
- 执行 `ws <URL>` 后，进入监听模式
- 用户可随时输入 `stop` 或 `结束` 停止监听
- 停止后输出完整报告

**输出报告格式**：
```markdown
# <页面> - WebSocket 监听报告

> 监听时间：YYYY-MM-DD HH:mm:ss
> 监听 URL：<URL>
> 监听时长：XX 秒

## 连接统计
| 指标 | 数值 |
| --- | --- |
| 总连接数 | X |
| 活跃连接数 | X |
| 重连次数 | X |
| 错误数 | X |

## 连接详情
| 连接 ID | URL | 状态 | 消息数 | 数据量 |
| --- | --- | --- | --- | --- |
| ws_1 | wss://api.example.com/ws | 活跃 | 120 | 45KB |

## Perps BBO 数据（如适用）
| 指标 | 数值 |
| --- | --- |
| 最新 buy1 | XXX |
| 最新 sell1 | XXX |
| 更新次数 | X |
| 价格变化次数 | X |

## 最近消息（最后 10 条）
| 时间 | 方向 | 类型 | 内容摘要 |
| --- | --- | --- | --- |
| 17:30:05 | 收到 | l2Book | {"coin":"BTC","levels":[[...]]} |

## 问题检测
- ✅ 连接稳定，无断连
- ⚠️ 消息更新频率过高（> 10次/秒）
- ❌ 检测到 3 次错误
```

### 1.3）analyze 指令 - 综合分析

**用途**：同时采集性能数据和 WebSocket 数据，全面诊断页面问题。

**执行流程**：
1. 连接浏览器（Chrome DevTools MCP）
2. 注入性能采集脚本（页面加载前）
3. 按需注入 WebSocket 监听脚本（仅在需要图表数据时）
4. 导航到目标 URL
5. 等待页面加载完成
6. 采集性能指标
7. 监听 WebSocket 消息（仅在需要时，默认 10 秒）
8. 输出综合分析报告

**输出报告格式**：
```markdown
# <页面> - 综合分析报告

> 分析时间：YYYY-MM-DD HH:mm:ss
> 分析 URL：<URL>

## 一、性能分析
（同 perf 指令的输出）

## 二、WebSocket 分析
（同 ws 指令的输出）

## 三、问题汇总
| 类型 | 问题 | 优先级 | 建议 |
| --- | --- | --- | --- |
| 性能 | LCP 超标 (3.2s) | P0 | 优化首屏资源加载 |
| WS | 重连次数过多 (5次) | P1 | 检查网络稳定性 |
```

## 2）冒烟测试（功能 + 性能一体化）

### 触发指令
| 指令 | 说明 |
| --- | --- |
| `@<用例文件> checklist` | 生成并输出可勾选清单，自动落盘 |
| `@<checklist文件> smoke <URL>` | 按 checklist 文件执行冒烟测试 
| `smoke <URL>` + 用户指令 | 按用户指令执行冒烟测试 |
| `smoke <URL>` | 等待用户提供指令或用例 |

### 核心逻辑
**在执行业务功能验证（UI 自动化）的同时，后台静默采集交互性能数据。性能指标达标是冒烟通过的必要条件。**

### 执行输入优先级（重要）

```
优先级从高到低：

1. 用户直接指令（最高优先级）
   └── 用户说："点击「最优价」按钮，然后选择「对手价1」"
   └── 直接按用户指令执行，无需用例文件

2. Checklist 文件
   └── @<checklist文件> smoke <URL>
   └── 按 checklist 文件中的操作步骤执行

3. 用例文件
   └── @<用例文件> smoke <URL>
   └── 按用例文件的步骤执行

4. 混合模式（用例文件 + 用户指令）
   └── 用户指令覆盖/补充用例文件
   └── 以用户最新指令为准

5. 无任何输入
   └── smoke <URL> 后等待用户输入
   └── 提示："请提供测试步骤或用例文件"
```

### 执行输入判断规则

| 场景 | 判断条件 | 执行逻辑 |
| --- | --- | --- |
| 用户给了直接指令 | 用户消息包含操作动词（点击、输入、选择、验证等） | **按用户指令执行** |
| 引用了 checklist 文件 | 消息以 `@<文件>` 开头，且文件路径包含 `checklist` 或文件名以 `-Checklist.md` 结尾 | 按 checklist 文件执行 |
| 引用了用例文件 | 消息以 `@<文件>` 开头，且为用例文件 | 按用例文件执行 |
| 两者都有 | 引用文件 + 额外指令 | 用户指令优先，覆盖文件 |
| 都没有 | 仅 `smoke <URL>` | 等待用户输入指令 |

### 执行模式
- ⚠️ **强制要求：必须使用调试模式** - 连接本地已打开的 Chrome 浏览器（通过 Chrome DevTools MCP，端口 9222）
  - ❌ **禁止**：使用 Playwright MCP 或其他会创建新窗口的工具
  - ✅ **必须**：使用 `mcp_chrome-devtools-9222_*` 系列工具
- **核心心法**：把 Agent 从一个"只会按剧本演戏的演员"变成一个"懂得随机应变的副驾驶"
- **执行原则**：先看路（Check State），再踩油门（Action），遇到红灯（钱包插件弹窗）时知道喊你一声

### 执行流程（smoke 指令）

> 详细的工具使用规范和工作流见 **chrome-mcp.md**

**详细流程**：
1. **启动自动确认脚本**（必须首先执行）：
   - 使用终端命令后台运行：`"/path/to/docs/scripts/auto-confirm.sh" &`
   - 确认脚本启动成功（检查标志文件 `/tmp/cursor-mcp-auto-confirm.flag` 存在）
   - **作用**：自动确认后续所有 MCP 工具调用弹窗，实现全自动执行
2. **确定执行步骤来源**（用户指令 > checklist 文件 > 用例文件 > 等待输入）
3. **连接浏览器**：使用 Chrome DevTools MCP 连接浏览器
4. **注册脚本（在导航前）**：
   - 使用 `Page.addScriptToEvaluateOnNewDocument` 注册性能采集脚本
   - 按需注册 WebSocket 监听脚本（仅当测试涉及图表数据时）
   - **关键**：必须在导航到 URL 之前完成脚本注册
5. **导航到 URL**：使用 `browser_navigate` 导航到目标 URL
6. **验证脚本注入**：
   - 等待页面加载完成
   - 检查脚本是否成功注入
   - 如果失败，自动刷新并重试（最多 2 次）
7. **循环执行**：**Check State → Action → Check State**
   - 遇到阻碍时：记录错误信息，跳过当前步骤，继续执行后续步骤
   - 不暂停等待：执行过程中不暂停等待用户手动操作
8. **集中处理异常步骤**：所有正常步骤执行完成后，尝试重新执行跳过的步骤
9. **生成一体化报告**（功能 + 性能 + 异常处理结果）
10. **停止自动确认脚本**：
    - 使用终端命令：`"/path/to/docs/scripts/stop-auto-confirm.sh"`
    - 或删除标志文件：`rm -f /tmp/cursor-mcp-auto-confirm.flag`

### Checklist 输出规则（checklist 指令）
当用户发送 `@<用例文件> checklist` 时，生成并输出清单，不执行：
1. **提取所有操作步骤**：从用例中提取涉及点击、输入、选择等操作的条目
   - ⚠️ **必须覆盖所有用例场景**：包括用例文件中的所有测试场景（P0/P1/P2 全部覆盖），不得遗漏任何场景
2. **转化为可执行清单**：每条一行，采用"序号. 操作步骤"的格式，简短描述
3. **聚焦"能不能用"**：冒烟测试验证功能是否可用
   - ⚠️ **重要说明**："聚焦能不能用"指验证功能可用性，**不代表可以忽略用例文件中的任何测试场景**，必须覆盖所有场景
4. **按操作顺序排列**：从入口到结束，按用户实际操作顺序排列
5. **输出到聊天窗口**：标准 Markdown checkbox 列表（可直接复制到飞书/Notion）
6. **自动落盘**：保存到 `docs/testcases/checklist/YYYY-MM-DD_<模块>-<测试主题>-Checklist.md`

**文件格式**：
- 文件内容按 `docs/specs/checklist.md` 第 10 节标准格式
- 包含：生成时间、来源用例、前置条件、操作步骤、关键断言点
- 文件内容可直接渲染为 Markdown，禁止最外层代码块包裹

**注意**：checklist 仅在用户请求时生成，smoke 执行时不自动生成 checklist，而是直接按用户指令、checklist 文件或用例文件执行。

### 冒烟测试通过标准
- **功能验证**：所有步骤执行成功，预期结果全部通过
- **性能指标**：Core Web Vitals（LCP、FID、CLS）必须全部达标
- **综合判定**：功能 + 性能缺一不可

### 全自动执行模式
> 详细的错误跳过与集中处理机制见 **chrome-mcp.md 第5节**

- **全自动执行**：执行过程中不暂停等待用户手动操作，系统自动完成所有操作和验证
- **错误跳过机制**：遇到阻碍时自动跳过该步骤，记录错误信息，继续执行后续步骤
- **集中处理异常**：所有正常步骤执行完成后，集中处理之前跳过的异常步骤
- **异常报告**：测试完成后生成异常处理报告，区分"已自动修复"和"需人工处理"的步骤

### 自动确认脚本管理
> 用于自动确认 Cursor MCP 工具调用弹窗，实现完全无人工干预

**脚本位置**：`docs/scripts/auto-confirm.sh`、`docs/scripts/stop-auto-confirm.sh`

**工作原理**：
- 通过标志文件 `/tmp/cursor-mcp-auto-confirm.flag` 控制脚本启停
- 仅在 Cursor 窗口激活且检测到 MCP 确认弹窗时才自动按 Enter
- 不影响其他程序的正常操作

**自动启停规则**：
1. **测试开始时**：Agent 在执行冒烟测试前，自动在后台启动 `auto-confirm.sh`
2. **测试结束时**：Agent 在输出测试报告后，自动调用 `stop-auto-confirm.sh` 停止脚本
3. **手动停止**：用户可随时运行 `pkill -f auto-confirm.sh` 或删除标志文件停止脚本

**使用注意**：
- 首次运行需授权 macOS "辅助功能" 权限给终端应用
- 脚本仅在 Cursor 是当前激活窗口时生效，切换到其他程序不会触发自动 Enter

### 批处理执行模式（推荐）
> 减少 Cursor 工具确认次数，提升执行效率。详见 **chrome-mcp.md 第5.5节**

**执行原则**：
1. **场景级批量执行**：将同一场景内的多个步骤视为一个执行单元，连续执行
2. **减少中间检查**：只在关键节点（场景开始/结束）进行状态检查
3. **容错优先**：单个步骤失败不终止流程，记录错误后继续执行
4. **最终汇总**：所有场景执行完成后，统一输出成功/失败/跳过的场景列表

**Cursor 确认优化**：
- 在 Cursor Settings → Features → MCP 中开启 "Auto-approve" 可减少确认弹窗
- 使用 `Cmd/Ctrl + Enter` 可快速确认工具调用

### 执行进度显示
- **简要说明测试内容**：在执行前简要说明正在测试的内容（如"正在测试限价单最优价格BBO功能"）
- **显示执行进度**：使用进度表示（如 `1/6` 或 `步骤 1/6`）显示当前执行进度
- **进度更新时机**：每完成一个步骤后更新进度
