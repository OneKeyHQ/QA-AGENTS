## 4）A. 通用检测维度（默认必须覆盖）
1. 输入与表单（Input & Forms）
   - 边界：空 / 空格 / 超长 / 特殊字符 / Emoji
   - 格式：精度、负数、千分位粘贴、自动 Trim
   - 安全：XSS / 注入尝试、敏感词过滤
2. 列表与展示（List & Display）
   - 空状态、分页 / 加载、排序 / 筛选
   - 实时刷新（WebSocket / 轮询）
   - 多语言 / 暗黑模式 / 长文本截断
3. 状态流转与逻辑（State & Logic）
   - 状态闭环（Pending / Success / Fail）
   - 断网 / 杀进程恢复
   - 并发点击、防抖、权限 / 未登录
4. 数值与资产计算（Calculation & Assets）
   - 精度 / 科学计数法 / 最小单位
   - 前端展示 vs 实际扣除一致性
   - 临界点（刚好够 / 刚好不足）
5. 网络与外部依赖（Network & External）
   - 弱网 / 超时 / 重试
   - 404 / 500 / 空数组容错
   - 权限拒绝（相机 / 剪贴板）引导
6. 性能（Performance）
   - 高频操作响应
   - 长列表 FPS
   - 页面反复进入内存释放
   - **Web 端专项**（见 `docs/specs/performance.md`）
7. 兼容性（Compatibility）
   - 小屏 / 刘海屏 / 折叠屏
   - 系统字体缩放
8. 安全（Security）
   - 敏感信息掩码
   - 关键操作二次确认 / 生物识别

## 5）B. 模块垂直深度映射（按识别模块自动引入）
### 钱包
- 状态机：创建/导入/备份校验/解锁/锁定/切链/切账户/恢复
- 资金一致性：资产页/详情页/交易记录三处一致；隐藏资产
- 权限/安全：助记词/私钥页截屏限制、剪贴板清除、导出权限、biometric 失败分支
- 链上不确定性：多 RPC 切换、链切换中断恢复、地址派生一致性（BIP44/49/84/86 等）
- 风控：高风险钱包提示、观察钱包 vs 冷钱包能力差异限制
- 可观测：错误码、导入失败原因可解释（格式/校验和/路径）
- 性能：大量资产/NFT 加载、缓存与刷新策略

### 转账（Transfer）

#### 核心功能覆盖（必须测试项）
生成转账相关链的测试用例时，必须覆盖以下功能点：

**软件钱包测试用例应包含**：
- (a) 主币转账
- (b) 代币转账（含普通代币、免 Gas 代币）
- (c) 私钥转账（如适用）
- (d) 私钥导入导出
- (e) DAPP 交易（连接、签名、合约交互）

**硬件钱包测试用例应包含**（仅输出以下三项，其他功能不输出）：
- (a) 主币转账
- (b) 代币转账（含普通代币、免 Gas 代币）
- (c) DAPP 交易（连接、签名、合约交互）

**硬件钱包不支持功能处理规则**：
- 如果某个功能在硬件钱包上不支持，不需要输出详细的不支持验证用例（如 Contract Token 转账不支持、Swap 不支持等）
- 仅在功能列表或测试概览中简单备注即可

#### 转账测试规则引用

转账相关的详细测试规则（金额测试规则、硬件钱包签名验证、硬件转账用例生成等）统一记录在 `docs/rules/transfer-chain-rules.md`（转账链规则文档）。

生成转账测试用例时，必须参考该文档中的规则：
- 金额转账测试规则（最大值、最小值、中间值）
- 硬件钱包签名流程验证规则
- 硬件钱包转账测试用例简化规则
- 硬件转账用例生成规则（主币和代币各 3 条用例）
- 各链的具体规则（最小转账金额、账户最低余额、Gas 规则、Memo 规则、硬件钱包支持情况等）

#### 钱包类型分组规则（强制）
**核心原则**：生成转账相关链的测试用例时，必须将软件钱包和硬件钱包的测试用例分开显示，输出为两篇独立的文档。

**分组方式**：
1. **软件钱包测试用例文档**（必须独立输出）：
   - 文件名：`YYYY-MM-DD_Wallet-<链名>-软件钱包测试.md`
   - 包含：主币转账、代币转账、私钥转账、私钥导入导出、DAPP 交易等相关功能点
   - 所有软件钱包相关功能集中在一个文档中

2. **硬件钱包测试用例文档**（必须独立输出）：
   - 文件名：`YYYY-MM-DD_Wallet-<链名>-硬件钱包测试.md`
   - 包含：主币转账、代币转账、DAPP 交易（仅此三项，其他功能不输出）
   - 不支持的功能只需简单标注"不支持"，无需详细用例
   - 所有硬件钱包相关功能集中在一个文档中

**分组理由**：
- 便于测试执行：同一类型钱包的功能集中，减少切换成本
- 便于自动化：同一类型钱包的测试逻辑相似，便于统一 Mock 和断言
- 便于维护：功能按钱包类型组织，结构清晰，易于定位和修改

**特殊情况**：
- 如果功能不区分钱包类型（如 Token 管理、余额查询），可独立分组或放在软件钱包文档中
- 如果功能仅支持一种钱包类型，按实际支持类型分组即可

#### 金额转账测试规则（强制）

所有涉及金额转账的用例必须覆盖最大值、最小值以及中间值。

**详细规则**：金额转账测试的详细规则（最大值、最小值、中间值的具体定义和计算方式）统一记录在 `docs/rules/transfer-chain-rules.md`（转账链规则文档）。

**规则维护**：
- 不同链的转账规则（最小转账金额、账户最低余额、Gas 规则等）统一记录在 `docs/rules/transfer-chain-rules.md`
- 生成用例时自动引用链规则文档中的规则
- 发现新规则或规则变更时，及时更新链规则文档

#### 转账模块通用规则
- 状态机：构建→签名→广播→Pending→Success/Fail→加速/取消/替换（Replace-by-fee/Nonce）
- 资金流：可用余额/冻结、Max 扣 Gas、手续费展示一致性、交易记录生成时机
- 地址/规则：合约地址拦截/提示、自转自、Memo/Tag 校验、ENS/域名解析
- 链上不确定性：Gas 波动、RPC 不一致、链重组导致状态回滚、广播失败重试
- 风控：黑名单地址、风险合约提示、大额二次确认、频率限制
- 可观测：失败原因必须落到可验证字段（error code + 用户文案 + hash/nonce）

### Swap
- 状态机：询价→路径→授权(Approve/Permit)→Swap→Pending→Success/Fail→回滚提示
- 资金流：授权额度变化、兑换前后余额对账、手续费/滑点/price impact 展示一致
- 链上不确定性：报价过期重算、同块抢跑导致滑点失败、nonce 冲突
- 代币兼容：通缩/税费币、rebase、黑名单 token、非标准 decimals
- 风控：高滑点/高 price impact 强提醒；目标合约风险提示；最小收到保护
- 数据源：路由/报价来源标识、刷新倒计时、WS/轮询策略一致
- 可观测：失败类型可分类（授权失败/余额不足/滑点/路由不可达/链错误）
- **核心测试路线**：详见 `docs/rules/swap-rules.md`（报价测试、构建订单测试、手续费测试、历史记录测试）
- **网络特性参考**：详见 `docs/rules/swap-network-features.md`（网络类型、主币信息、授权要求、交易费单位等），生成 Swap 测试用例时必须参考网络特性表

### Market
- 数据一致性：K线/列表/详情价格一致；多源差异策略（取中位/优先源/降级）
- 状态机：WS 断连→重连→补数据→防闪烁；切币/切交易对/切法币
- 极端行情：精度丢失、科学计数法、涨跌幅异常、停牌/新币无数据
- 性能：长列表虚拟化、K 线交互帧率、缓存/预加载
- 可观测：数据源降级提示、请求失败可恢复路径
- **限价单特殊逻辑**（CowSwap 渠道，支持网络：Ethereum、Arbitrum、Base）：
  - FromToken 不支持直接出售 ETH，只能换成 WETH（需要验证 ETH → WETH 自动转换逻辑）
  - 订单有效期设置：支持 5分钟/30分钟/1小时/1天/3天/7天/1个月（需要验证各选项功能及到期处理）
  - 部分成交：支持启用/禁用（需要验证部分成交和完全成交的不同处理逻辑）
  - 特殊情况：Ethereum USDT 到 ETH 需要二次授权（USDT → ETH），USDT 到其他代币不需要二次授权

### DeFi
- 状态机：授权→存入→计息/出块→领取→赎回（含锁定期/惩罚/排队）
- 资金流：本金/收益拆分、APR/APY 口径、T+N 到账、奖励延迟与快照
- 合约风险：暂停/升级/拒绝、额度管理、claim 多次幂等
- 链上不确定性：数据延迟、RPC 分叉、pending 与到账延迟的用户解释
- 风控：高风险池提示、最低存入限制、最大额度限制、重复操作保护
- 可观测：收益计算公式与数据来源可追溯（字段/时间点/快照）
- **DeFi Borrow 借贷协议规则**：详见 `docs/rules/borrow-defi-rules.md`

### Perps（合约）
- 状态机：开仓→加仓→减仓→平仓→反手；挂单/撤单；TP/SL 生效/失效；切币，默认 USDC 法币
- 资金流：保证金/可用余额/占用保证金、资金费率结算扣款、强平后残值
- 风控：维持保证金率预警、最大杠杆/最大仓位、爆仓保护、价格保护
- 数据源：盘口/成交/仓位 WS 更新节流、防跳动；断连重连一致性；接入的 hyperliquid
- 链上/链下：撮合与链上交互混合时的"确认口径"、失败回滚解释
- 可观测：关键字段必须可断言（entry/mark/liquidation/margin/fee/funding）
- **核心测试规则**：详见 `docs/rules/perps-rules.md`（限价单最优价 BBO 规则、收藏功能规则等）

### DApp
- 状态机：connect→请求权限→签名/交易→用户确认/拒绝→回到 dapp 的回传
- 安全：origin 展示、权限范围、钓鱼/相似域名、盲签阻断（解析失败时强警示）
- 兼容：多 tab、EIP-6963、多钱包冲突、网络切换请求处理
- 链上不确定性：签名弹窗中断、返回后恢复、重复请求幂等
- 可观测：签名内容解析结果可验证（method/params/金额/收款地址/合约地址）

### 返佣
- 状态机：绑定→生效→累计→结算→发放→冻结/解冻
- 资金流：累计收益口径、历史回溯修正、T+N、精度与四舍五入
- 风控：封禁后冻结与解冻补发策略
- 数据一致性：跨端一致（Web/App），邀请人跟被邀请人奖励能对应
- 可观测：结算失败原因可追踪（订单缺失/链上查不到/延迟/风控）

### 通知
- 状态机：生成→推送→到达→点击跳转→已读；离线堆积→去重
- 路由：Deep link 失效兜底（打开列表/详情）；跳转参数校验
- 数据一致性：消息与业务状态一致（交易成功必须能定位到 hash 详情）
- 可观测：重复/丢失的判定与日志

### 风控
- 设备：root/越狱/模拟器/篡改检测
- 交易拦截：黑名单/高危合约/大额二次确认/频率限制/验证码
- 隐私：日志脱敏、缓存加密、截图/录屏策略
- 可观测：拦截原因必须"可验证"（规则 id + 命中字段 + 用户提示）

### NFT
- 状态机：加载→缓存→刷新→隐藏垃圾→展示详情→发送/签名
- 数据源：Metadata/IPFS 超时降级、图片/视频策略、占位与重试
- 风控：钓鱼 NFT 自动隐藏/提示、恶意链接拦截
- 性能：长列表滚动、缩略图缓存、分页

### 7.0 测试用例文件保存路径规范（强制）

**核心规则**：所有生成的测试用例文件必须统一保存到 `docs/testcases/cases/` 目录下，并按模块自动分类到对应的子目录。

**路径规范**：
- **测试用例文件**：`docs/testcases/cases/<模块目录>/YYYY-MM-DD_<模块>-<测试主题>.md`
- **Checklist 文件**：`docs/testcases/checklist/YYYY-MM-DD_<模块>-<测试主题>-Checklist.md`
- **性能报告文件**：`docs/testcases/performance/YYYY-MM-DD_<模块>-冒烟测试.md`
- **API 测试用例文件**：`docs/testcases/api/{collection}-Apifox-TestCases.json`

**模块自动分类规则**：
1. 从文件名中提取模块名（文件名格式：`YYYY-MM-DD_<模块>-<主题>.md`，提取第一个 `-` 之前的内容）
2. 根据模块名称映射表自动分类到对应子目录
3. 如果无法识别模块，使用 `other/` 目录作为兜底

**模块名称映射表**：
| 模块名（文件名中） | 目录名 | 说明 |
|-----------------|--------|------|
| `Wallet` / `钱包` | `wallet/` | 钱包模块 |
| `Transfer` / `转账` | `transfer/` | 转账模块 |
| `Swap` | `swap/` | Swap 模块 |
| `Market` / `市场` | `market/` | Market 模块 |
| `DeFi` / `Defi` | `defi/` | DeFi 模块 |
| `Perps` / `合约` | `perps/` | Perps 模块 |
| `DApp` / `Dapp` / `dApp` | `dapp/` | DApp 模块 |
| `返佣` / `Referral` | `referral/` | 返佣模块 |
| `通知` / `Notification` | `notification/` | 通知模块 |
| `风控` / `Security` | `security/` | 风控模块 |
| `NFT` | `nft/` | NFT 模块 |
| 其他未匹配 | `other/` | 兜底目录 |

**匹配规则**：
- 模块名匹配不区分大小写（`Wallet` = `wallet` = `WALLET`）
- 优先匹配英文模块名，如果匹配不到再匹配中文模块名
- 中文模块名需要完整匹配（"钱包"匹配，但"钱包管理"不匹配）

- 生成用例时自动识别模块并创建对应的子目录（如果不存在）
- 确保所有测试用例按模块集中管理，便于查找、维护和版本控制
- 生成用例时自动创建文件，无需用户手动指定路径

### 7.1 用例标题（单行）
用例标题：`<模块名或组合> - <测试主题>`

### 7.2 按测试场景分组
示例：
1. 未登录状态
2. 已登录主流程
3. 弱网与异常
4. 风控拦截

### 7.3 每个场景输出表格（固定格式）
使用 **Markdown 表格**，单元格内多行使用 `<br>` 标签换行。

表头固定为：
`| 优先级 | 自动化层级 | 输入数据 | 操作步骤 | 预期结果 (含自动化断言) |`

规则：
- 同场景可多行（P0 / P1 / P2）。
- 输入数据 / 操作步骤 / 预期结果必须编号（如：1. xxx<br>2. xxx）。
- **关键格式**：表格单元格内多行内容使用 `<br>` 标签分隔。
- 不同测试场景禁止合并为同一张表。

### 7.3.1 钱包账户类型分组规则（强制）

**核心原则**：新链接入按钱包账户类型（软件钱包/硬件钱包）分成两条用例输出，同一类型的所有功能测试集中在一起。

**分组方式**：

1. **软件钱包分组**（必须集中）：
   - 软件钱包主币转账
   - 软件钱包代币转账（含普通代币、免 Gas 代币）
   - 软件钱包 dApp 连接与交易
   - 软件钱包私钥/助记词导出
   - 软件钱包账户管理

2. **硬件钱包分组**（必须集中，仅输出以下三项）：
   - 硬件钱包主币转账
   - 硬件钱包代币转账（含普通代币、免 Gas 代币）
   - 硬件钱包 dApp 连接与交易
   - **不支持功能处理**：如果某个功能在硬件钱包上不支持，只需简单标注"不支持"，无需输出详细用例


**分组理由**：
- 便于测试执行：同一类型钱包的功能集中，减少切换成本
- 便于自动化：同一类型钱包的测试逻辑相似，便于统一 Mock 和断言
- 便于维护：功能按钱包类型组织，结构清晰，易于定位和修改

**特殊情况**：
- 如果功能不区分钱包类型（如 Token 管理、余额查询），可独立分组
- 如果功能仅支持一种钱包类型，按实际支持类型分组即可

### 7.6 交付格式（Markdown 模式）
#### 针对 Chat 聊天窗口输出（为了方便复制）

## 8）生成前自动执行：规则维护（强制）

### 规则维护自动化流程（必须执行）

生成测试用例前，必须自动执行以下步骤：

#### 步骤 1：识别新需求/新规则
- 分析用户输入的需求，判断是否为新的功能点或规则
- 识别所属模块（如 Perps、Swap、Transfer 等）
- 识别功能名称（如"收藏功能"、"限价单最优价"等）

#### 步骤 2：检查并创建/更新需求文档
- **检查路径**：`docs/requirements/<模块>-<功能名>.md`
- **如果不存在**：自动创建需求文档
  - 文件名格式：`<模块>-<功能名>.md`（如 `Perps-收藏功能.md`）
  - 包含：需求背景、功能描述、业务规则、已知风险、关联资源、变更记录
- **如果存在**：检查是否需要更新（新增功能点、规则变更等）
  - 在变更记录中追加更新内容

#### 步骤 3：检查并创建/更新规则文档
- **检查路径**：`docs/rules/<module>-rules.md`（如 `docs/rules/perps-rules.md`）
- **如果不存在**：自动创建规则文档
  - 包含：功能规则、测试规则、边界条件、变更记录
- **如果存在**：检查是否需要更新（新增规则、规则变更等）
  - 在对应功能章节中添加新规则
  - 在变更记录中追加更新内容

#### 步骤 4：更新 qa-rules.md
- **如果是新模块**：在 `qa-rules.md` 的"模块垂直深度映射"章节中添加模块章节
- **如果是已有模块的新功能**：检查模块章节是否需要更新
- **添加规则文档引用**：在模块章节中添加规则文档引用（如：`详见 docs/rules/<module>-rules.md`）

#### 步骤 5：生成测试用例
- 引用最新创建/更新的需求文档和规则文档
- 确保测试用例覆盖规则文档中的所有规则

### 规则文档组织原则
- **通用规则**：优先把规则固化到 `docs/qa-rules.md`
- **模块专项规则**：按模块拆分到 `docs/rules/<module>-rules.md`
- **需求文档**：新增或更新 `docs/requirements/<模块>-<功能名>.md`
- **执行顺序**：先创建/更新需求文档和规则文档，再生成测试用例

## 9）生成后自动执行：合并 & 去重（强制）
### 合并规则
同一测试场景下：
- 输入数据一致，或
- 操作步骤一致，仅校验点不同
则必须合并为一条（或一行）用例描述，避免重复。

### 去重规则
- 删除语义重复的预期
- 删除隐含重复步骤
- 同一校验点只保留一次

### 禁止合并
- 不同测试场景
- 不同业务状态
- 不同优先级（P0 不得与 P1/P2 合并）

### 结果自检
- 合并后仍可人工执行
- 可拆为 UI / API / E2E 自动化
- 覆盖资金 / 风控 / 状态切换
- 确认表格中使用了 `<br>` 且无超链接样式
