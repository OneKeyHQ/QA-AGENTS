# QA 项目技能文档（Skill）

> 本项目让测试用例生成与规则维护"落在仓库里"，不依赖用户粘贴 Prompt。

---

## 🎯 核心能力

| 能力 | 触发指令 | 输出 |
| --- | --- | --- |
| 生成测试用例 | `生成测试用例` | 结构化用例表格（标题 → 场景 → 表格）+ 自动落盘 |
| 冒烟测试清单 | `@<用例文件> checklist` | 精细化步骤 Checkbox 列表 + 自动落盘 |
| 冒烟测试执行 | `smoke <URL>` | chrome 功能验证 + 性能采集一体化报告 |
| **E2E 脚本生成** | `@<文件> e2e` | Playwright 测试脚本 + 自动落盘 |
| **性能分析** | `perf <URL>` | 仅采集性能数据（不执行操作） |
| **WS 监听** | `ws <URL>` | 仅监听 WebSocket 数据 |
| **综合分析** | `analyze <URL>` | 性能 + WS 综合分析 |

---

## 🚀 使用方法

### 1. 生成测试用例

**输入**：需求文档（`.doc` / `.md`）或直接描述需求

**指令**：
```
@<需求文档> 生成测试用例
```

**输出**：
- 聊天窗口：完整用例（封装在 `markdown` 代码块中，便于一键复制）
- 自动落盘：`docs/testcases/cases/YYYY-MM-DD_<模块>-<主题>.md`

**示例**：
```
@Perps 限价单最优价格需求.doc 生成测试用例
```

---

### 2. 冒烟测试清单（提测验收）

**输入**：已生成的测试用例文件

**指令**：
```
@<用例文件> checklist
```

**输出**：
- **自动落盘**：`docs/testcases/checklist/YYYY-MM-DD_<模块>-<测试主题>-Checklist.md`


**文件格式**：
- 包含：生成时间、来源用例、前置条件、操作步骤、关键断言点
- 标准 Markdown checkbox 格式，可直接用于冒烟测试执行

---

### 3. 冒烟测试执行（功能 + 性能一体化）

**核心逻辑**：在执行业务功能验证（UI 自动化）的同时，后台静默采集交互性能数据。**性能指标达标是冒烟通过的必要条件**。

**指令**：
| 指令 | 说明 |
| --- | --- |
| `smoke <URL>` + 用户指令 | 按用户指令执行（最高优先级） |
| `@<checklist文件> smoke <URL>` | 按 checklist 文件执行 |
| `@<用例文件> smoke <URL>` | 按用例文件执行 |
| `smoke <URL>` | 等待用户提供指令或用例 |

**执行优先级**：
```
1. 用户直接指令（最高）→ 直接按指令执行
2. Checklist 文件 → 按 checklist 步骤执行
3. 用例文件 → 按用例步骤执行
4. 混合模式 → 用户指令覆盖文件
5. 无任何输入 → 提示用户提供测试步骤
```

**执行流程**：
1. **确定执行步骤来源**：用户指令 > checklist 文件 > 用例文件 > 等待输入
2. **连接浏览器**：使用 Chrome MCP 连接浏览器
3. **注入性能采集脚本**：在页面加载前注入（无需手动操作）
4. **简要说明测试内容**：在执行前简要说明正在测试的内容（如"正在测试限价单最优价格BBO功能"）
5. **执行功能验证**：按用户指令、checklist 步骤或用例步骤执行 UI 自动化操作，显示执行进度（如 `1/6` 或 `步骤 1/6`）
6. **静默性能采集**：后台自动采集性能指标
7. **生成一体化报告**：功能验证结果 + 性能指标汇总（只显示测试概要、结果汇总、性能分析、截图、总结、问题）

**性能采集说明**：
- ✅ **全自动执行**：所有性能采集脚本由系统自动注入和执行
- ✅ **透明化采集**：性能采集在后台静默进行，不影响功能验证流程
- ✅ **按需注入 WS 脚本**：**只有涉及到图表数据相关的情况才会调用**，不要默认在所有模块测试中请求 WS 数据

**能力**：
- **功能验证**：基于用户指令、checklist 文件或用例文件执行
- **性能采集**：Core Web Vitals（LCP、FID、CLS）、加载性能、资源性能、网络性能、运行时性能
- **通过标准**：功能验证通过 **且** 性能指标达标，两者缺一不可

**输出**：
- 聊天窗口：一体化测试报告（Markdown 格式）
  - 报告内容：测试概要、测试结果汇总、性能分析、页面截图、测试总结、发现的问题（没有则空着）
  - 不输出详细测试结果
- 自动落盘：`docs/testcases/performance/YYYY-MM-DD_<模块>-冒烟测试.md`

**人工干预**：
- **仅在明显阻碍时提示**：只有在流程遇到明显阻碍（如元素找不到且重试失败、操作失败且无法自动恢复、页面异常导致无法继续等）时才暂停并提示用户
- **自动处理常见场景**：钱包插件弹窗、授权请求、密码输入等常见场景应自动处理或跳过，不暂停执行

**示例**：
```
smoke https://app.onekey.so
```
```
@docs/testcases/cases/2025-12-31_Perps-限价单最优价格BBO.md smoke https://app.onekey.so/perps
```
```
@docs/testcases/checklist/2026-01-04_Perps-限价单最优价格BBO-Checklist.md smoke https://app.onekey.so/perps
```

**性能指标覆盖**：
- **Core Web Vitals**：LCP、FID、CLS（必须达标）
- **加载性能**：FCP、TTI、TBT
- **资源性能**：资源加载时间、资源大小
- **网络性能**：请求数量、请求耗时、失败率
- **运行时性能**：FPS、内存占用、CPU 使用率

**WebSocket 监听能力**（按需调用）：
- ✅ **按需调用原则**：只有涉及到图表数据相关的情况才会调用 WS 监听
- ✅ **不默认调用**：不要默认在所有模块测试中请求 WS 数据，即使模块是 Perps/Market
- ✅ **场景驱动**：根据测试用例内容判断是否需要 WS 监听（如 K线图、实时价格图表、BBO 价格图表等）
- ✅ **自动监听所有 WebSocket 连接**：仅在需要时，无需手动配置
- ✅ **自动提取 Perps BBO 数据**：buy1/sell1 价格自动提取（仅在需要时）
- ✅ **连接状态监控**：断连/重连自动检测（仅在需要时）
- ✅ **消息内容记录**：所有 WS 消息自动记录和解析（仅在需要时）
- ✅ **集成到测试报告**：WebSocket 数据自动包含在冒烟测试报告中（仅在需要时）

### 4. E2E 脚本生成（Playwright）

**输入**：Checklist 文件或测试用例文件

**指令**：
```
@<checklist文件> e2e
```
或
```
@<用例文件> e2e
```

**输出**：
- 自动落盘：`e2e/tests/{模块名}-{功能名}.spec.ts`

**可选参数**：
| 参数 | 说明 |
| --- | --- |
| `--full` | 生成完整模式（包含 P0/P1/P2 所有场景） |
| `--no-perf` | 不生成性能测试 |

**运行生成的脚本**：
```bash
cd e2e && npx playwright test {脚本名}.spec.ts
```

**示例**：
```
@docs/testcases/checklist/2026-01-04_Market-Token收藏取消收藏-Checklist.md e2e
```

**详细文档**：`docs/skills/playwright-generator/SKILL.md`

### 5. 独立分析指令（性能 / WebSocket）

> 当你只想分析页面性能或 WebSocket 数据，而不需要执行完整的冒烟测试时，使用以下独立指令。

**指令对比**：
| 指令 | 功能验证 | 性能采集 | WS 监听 | 适用场景 |
| --- | --- | --- | --- | --- |
| `smoke <URL>` | ✅ | ✅ | ✅（仅图表数据相关） | 完整冒烟测试 |
| `perf <URL>` | ❌ | ✅ | ❌ | 快速检查页面加载性能 |
| `ws <URL>` | ❌ | ❌ | ✅ | 调试实时数据推送 |
| `analyze <URL>` | ❌ | ✅ | ✅ | 全面诊断页面问题 |

**perf 指令**：
```
perf https://app.onekey.so/perps
```
- 仅采集性能数据，不执行任何操作
- 输出：Core Web Vitals、加载性能、资源分析、慢资源 Top 5

**ws 指令**：
```
ws https://app.onekey.so/perps
```
- 仅监听 WebSocket 连接和消息
- 适用模块：Perps、Market（有实时数据的模块）
- 输出：连接统计、BBO 数据、消息记录
- 交互式监听：输入 `stop` 或 `结束` 停止监听

**analyze 指令**：
```
analyze https://app.onekey.so/perps
```
- 同时采集性能 + WebSocket 数据
- 输出：综合分析报告（性能 + WS + 问题汇总）

## 📋 用例输出规范

### 格式声明
- **输出格式**：标准 Markdown（`.md`）
- **换行符**：表格内多行使用 `<br>` 标签
- **适用平台**：Markdown Preview Enhanced、GitHub、GitLab、Notion、Jira、飞书
- **优势**：标准 Markdown 语法，插件可正确渲染换行

### 结构
```
# <模块> - <测试主题>
> 生成时间：YYYY-MM-DD

## 测试场景列表

### 1. 场景名称
| 优先级 | 自动化层级 | 输入数据 | 操作步骤 | 预期结果 (含自动化断言) |
| --- | --- | --- | --- | --- |
| P0 | E2E | 1. ...
2. ... | 1. ...
2. ... | 1. ...
2. ... |

### 2. 场景名称
...

## 自动化实施方案（P0/P1）
...
```

### 优先级定义
| 优先级 | 覆盖范围 |
| --- | --- |
| P0 | 资金 / 安全 / 风控 / 主流程 / 状态切换 |
| P1 | 配置 / UI / 非主路径 |
| P2 | 低频 / 文案 / 容错 |

### 输出硬性约束（任何“生成测试用例”类请求都要满足）
- 禁止输出“模块感知分析/内部推理过程”（只做不说）。
- 按 `docs/qa-rules.md` 的结构输出：标题 → 场景分组 → 场景表格。
- 交付格式以 `docs/qa-rules.md` 为准；当规则要求“Cursor 专用版”时：
  - 不在聊天窗口直接渲染表格
  - 将全部交付内容封装在 **一个** `markdown` 代码块中
- 表格单元格内多行内容使用 `<br>` 标签分隔。
- 禁止使用“正常 / 符合预期 / 展示正确”等不可断言表达；预期必须可验证、可断言、可自动化。
- 禁止渲染超链接样式：URL/ID 仅以纯文本展示。

## ✅ Checklist 输出规范（精细化步骤清单）

### 核心目标
将用例模板转化为**连贯、可直接执行**的精细化步骤清单，每个步骤只包含 1 个具体动作，操作对象精准无歧义。

### 输出格式
```markdown
# <模块> - <主题> 冒烟测试 Checklist

## 前置条件
- [ ] 条件1：[具体状态描述]
- [ ] 条件2：[具体状态描述]

## 操作步骤
- [ ] 1. [具体操作，明确操作对象+动作] → [预期结果]
- [ ] 2. [具体操作，明确操作对象+动作] → [预期结果]
- [ ] 3. [具体操作，明确操作对象+动作] → [预期结果]
...

## 关键断言点
- [ ] [步骤X完成后] 断言描述
- [ ] [步骤Y完成后] 断言描述
```

### 步骤编写规则
| 规则 | 说明 | 示例 |
| --- | --- | --- |
| 每步1个动作 | 禁止合并多个动作 | ✅ `1. 输入密码` + `2. 点击登录` ❌ `输入密码并点击登录` |
| 操作对象精准 | 使用「」标注元素名称 | ✅ `点击页面底部的「提交」按钮` ❌ `点击相关按钮` |
| 输入值明确 | 使用 "" 标注具体值 | ✅ `输入"123456abc"` ❌ `输入密码` |
| 位置描述 | 明确元素在页面的位置 | ✅ `点击页面顶部右侧的「连接钱包」按钮` |
| 动作+对象 | 格式为"动作+位置+元素" | ✅ `在「价格输入框」中输入"100.05"` |

### 禁止事项
- ❌ 模糊表述：如"点击相关按钮"、"填写必要信息"
- ❌ 省略动作：如省略"返回上一页"、"确认弹窗"
- ❌ 合并步骤：如"输入密码并点击登录"
- ❌ 保留优先级：输出清单不体现 P0/P1/P2

### 规则
- 按实际操作顺序排列
- 覆盖所有点击/输入类动作
- 前一步输出为后一步前置条件

## 🔧 规则维护

### 规则文档组织原则

为了便于阅读和维护，规则文档采用模块化拆分：

1. **核心规则**：`docs/qa-rules.md`
   - 用例生成通用规则
   - 模块垂直深度映射
   - 输出格式要求

2. **模块专项规则**：`docs/rules/` 目录
   - 按模块拆分专项规则文档
   - 例如：`transfer-chain-rules.md`（转账链规则）
   - 便于独立维护和更新

3. **需求文档**：`docs/requirements/` 目录
   - 具体功能需求文档
   - 包含业务规则和测试要点

### 转账模块规则说明

**钱包类型分组**：
- 生成转账相关链的测试用例时，必须将软件钱包和硬件钱包分开显示
- 输出为两篇独立文档：
  - `YYYY-MM-DD_Wallet-<链名>-软件钱包测试.md`
  - `YYYY-MM-DD_Wallet-<链名>-硬件钱包测试.md`

**金额转账测试规则**：
- 必须覆盖最大值、最小值、中间值
- 最大值：钱包里的余额（用户自己的可用值）
- 最小值：根据具体链和代币规则（参考 `docs/rules/transfer-chain-rules.md`）
- 具体规则请查看 `docs/rules/transfer-chain-rules.md`

### 规则更新流程

当引入新需求/新规则时：
1. **通用规则**：更新 `docs/qa-rules.md`
2. **模块专项规则**：更新或创建 `docs/rules/<module>-rules.md`
3. **需求文档**：新增或更新 `docs/requirements/<topic>.md`
4. **生成用例时**：自动引用相关规则文档
