# Chrome MCP 工具使用规范

> 本节定义使用 Chrome DevTools MCP 进行自动化测试的工具规范。

---

## 1）连接浏览器（调试模式）

### 1.1 连接方式

**直接连接到已运行的 Chrome**（推荐）：
- 使用 Chrome DevTools MCP 连接到本地 Chrome 浏览器（端口 9222）
- 无需复制数据，直接使用当前浏览器的所有状态（钱包、登录、扩展等）

### 1.2 启动/检查脚本

```bash
# 检查 CDP 是否可用（连接模式）
./docs/scripts/start-mcp-chrome.sh

# 如需启动新实例（轻量模式，不复制数据）
./docs/scripts/start-mcp-chrome.sh --new

# 如需启动新实例（完整模式，复制数据）
./docs/scripts/start-mcp-chrome.sh --new --copy
```

### 1.3 前提条件

- **Chrome 需带调试端口启动**：确保 Chrome 启动时带 `--remote-debugging-port=9222` 参数
- **使用有数据的钱包**：确保连接的浏览器中已登录钱包且有测试数据

### 1.4 如何让 Chrome 默认带调试端口启动

**macOS**：创建别名或修改启动方式
```bash
# 方法1：使用别名（添加到 ~/.zshrc）
alias chrome-debug='/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222'

# 方法2：关闭现有 Chrome 后启动
pkill -f "Google Chrome"
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222 &
```

## 2）必须使用的工具

> 使用 `mcp_chrome-devtools-9222_*` 系列工具（Chrome DevTools MCP）

| 工具 | 用途 |
|-----|------|
| `list_pages` | 获取页面列表，验证连接 |
| `select_page` | 选择目标页面 |
| `navigate_page` | 导航到目标 URL |
| `wait_for` | 等待页面加载完成或关键元素出现 |
| `take_snapshot` | **核心工具**，获取页面快照用于状态检查 |
| `list_network_requests` | 获取网络请求数据 |
| `list_console_messages` | 获取控制台消息和性能指标 |
| `evaluate_script` | 执行 JavaScript 脚本（性能采集、数据获取） |

**`take_snapshot` 用途**：
- 操作前检查：获取页面快照，分析关键元素是否存在、页面是否可操作
- 操作后验证：验证操作是否成功、状态是否符合预期
- 人工干预检测：识别钱包插件弹窗、授权请求等需要人工操作的场景

## 4）状态检查工作流（Check State → Action → Check State）

1. **操作前**：使用 `take_snapshot` 获取页面快照
   - 分析快照中的元素（通过 `uid` 标识），确认目标元素是否存在
   - 检查页面状态（是否加载完成、是否有错误）
   - 检查是否有弹窗遮挡（钱包插件弹窗、授权请求等）
   - 自动处理常见弹窗（不暂停执行）
2. **执行操作**：确认状态无误后，使用 `uid` 执行 UI 操作（`click`、`fill` 等）
3. **操作后**：使用 `take_snapshot` 再次获取页面快照
   - 验证操作是否成功（元素状态变化、页面跳转等）
   - 检查是否有错误提示或异常状态
   - 确认页面状态符合预期

## 5）错误跳过与集中处理机制（Error Skip & Batch Retry）

> **核心原则**：全自动执行，无需人工确认。遇到阻碍时跳过并继续，最后集中处理异常步骤。

### 5.0 Cursor 工具调用确认机制（重要）

**Cursor 安全机制说明**：
- Cursor IDE 的 MCP 工具调用默认需要用户点击确认，这是 IDE 层面的安全机制
- **解决方案**：在 Cursor Settings → Features → MCP 中，可以设置"Auto-approve"模式，减少确认次数
- **替代方案**：如果无法关闭确认，可以在 Cursor 中按住 `Cmd/Ctrl` 并点击确认按钮，快速批量确认

**执行策略优化**（减少确认次数）：
- ✅ **批量操作**：将多个相关步骤合并为一次工具调用（如一次 `evaluate_script` 执行多个检查）
- ✅ **减少快照**：非关键步骤跳过中间状态检查，只在关键节点检查状态
- ✅ **批量断言**：在一次脚本执行中完成多个断言，而不是每个断言单独调用

### 5.1 执行模式

**全自动执行**：
- ✅ **不暂停等待**：执行过程中不暂停等待用户手动操作（Cursor 工具确认除外）
- ✅ **自动跳过**：遇到阻碍时自动跳过该步骤，记录错误信息，继续执行后续步骤
- ✅ **集中处理**：所有步骤执行完成后，集中处理之前跳过的异常步骤
- ✅ **错误容忍**：单个步骤失败不终止整体流程，继续执行后续步骤

### 5.2 错误检测与分类

1. **检测触发**：使用 `take_snapshot` 获取页面快照后，分析快照中的元素
2. **错误分类**：
   - **可自动处理错误**（自动处理，不记录）：
     - 钱包插件弹窗：尝试自动关闭或跳过
     - 授权请求：尝试自动处理或跳过
     - 密码输入：尝试自动填充或跳过
     - 常见弹窗：尝试自动关闭或跳过
     - 元素暂时不可见：等待后重试（最多 3 次，每次等待 1 秒）
     - 操作暂时失败：重试后继续（最多 3 次）
   
   - **需跳过错误**（记录错误，跳过步骤，继续执行）：
     - 元素找不到且重试失败（已尝试多种定位方式仍无法找到，已重试 3 次）
     - 操作失败且无法自动恢复（点击无响应、输入失败等，已重试 3 次）
     - 页面异常导致无法继续（错误提示、加载失败等，且无法自动恢复）
     - 任何导致当前步骤无法完成的问题

### 5.3 错误跳过机制

**执行流程**：
1. **检测到错误**：在执行步骤时检测到无法自动处理的错误
2. **记录错误信息**：
   ```
   错误记录格式：
   {
     "stepIndex": 3,
     "stepDescription": "点击「最优价」按钮",
     "errorType": "element_not_found",
     "errorMessage": "元素「最优价」按钮找不到，已尝试多种定位方式仍无法找到",
     "timestamp": "2026-01-06 10:30:15",
     "pageState": "快照信息或页面URL"
   }
   ```
3. **跳过当前步骤**：标记该步骤为"已跳过"，不执行后续操作
4. **继续执行**：立即继续执行下一个步骤，不暂停等待

### 5.4 集中处理异常步骤

**触发时机**：所有正常步骤执行完成后（包括成功和跳过的步骤）

**处理流程**：
1. **汇总异常步骤**：收集所有跳过的步骤，按执行顺序排列
2. **尝试重新执行**：
   ```
   For each 跳过的步骤:
     1. 使用 take_snapshot 检查当前页面状态
     2. 判断是否满足执行条件
     3. 如果满足，尝试重新执行该步骤
     4. 如果成功，标记为"已修复"
     5. 如果仍失败，标记为"需人工处理"
   ```
3. **生成异常报告**：
   ```
   ## 异常步骤处理结果
   
   ### 已自动修复
   - [步骤 3] 点击「最优价」按钮 ✅ 已修复
   
   ### 需人工处理
   - [步骤 5] 输入金额 "100" ❌ 元素仍无法找到
     - 错误信息：元素「金额输入框」找不到，已尝试多种定位方式仍无法找到
     - 建议操作：请检查页面是否正常加载，或手动完成该步骤后继续
   ```

### 5.5 批处理执行模式（Batch Execution Mode）

> **目标**：减少 Cursor 工具确认次数，提升执行效率。

**批处理策略**：

1. **场景批量执行**：
   - 将同一场景内的多个步骤视为一个执行单元
   - 在场景开始时获取一次快照，场景结束时验证一次
   - 场景内的中间步骤连续执行，不单独验证

2. **操作合并**：
   - 连续的点击操作：合并为一次执行，中间用短延时分隔
   - 输入 + 点击：合并为一次表单填写 + 提交操作
   - 多个验证：在一次 `evaluate_script` 中完成所有断言

3. **智能等待**：
   - 使用 `wait_for` 等待关键元素，而不是固定延时
   - 批量操作后统一等待页面稳定

**执行流程**：
```
For each 场景 in Checklist:
  1. [可选] 场景开始前快照检查
  2. 批量执行场景内的所有步骤（连续调用，不中断）
  3. 等待页面稳定（使用 wait_for 或短延时）
  4. 场景结束后统一验证结果
  5. 记录场景执行状态（成功/失败/跳过）
  6. 继续下一个场景（不论当前场景是否成功）
```

### 5.6 错误汇总与最终报告

**最终报告结构**：
```markdown
## 测试执行总结

### 执行概览
| 指标 | 数值 |
|------|------|
| 总场景数 | X |
| 成功场景 | X |
| 跳过场景 | X |
| 失败场景 | X |
| 总通过率 | XX% |

### 成功场景
- [场景 1] 观察列表测试 ✅
- [场景 2] 热门代币测试 ✅

### 跳过场景（已重试）
- [场景 3] 搜索功能测试 ⏭️ 
  - 跳过原因：搜索框元素未找到
  - 重试结果：仍失败

### 失败场景
- [场景 4] 数据一致性测试 ❌
  - 失败原因：API 返回数据与页面显示不一致
  - 详细信息：预期价格 100.05，实际显示 100.04

### 需人工处理
1. [场景 3] 搜索功能测试
   - 问题：搜索框元素定位失败
   - 建议：检查页面是否有遮罩层，或元素选择器是否变更
```

### 5.5 错误处理策略

**重试策略**：
- **元素查找**：最多重试 3 次，每次间隔 1 秒
- **操作执行**：最多重试 3 次，每次间隔 0.5 秒
- **页面等待**：最多等待 5 秒，每秒检查一次

**自动恢复策略**：
- **弹窗处理**：尝试查找并点击关闭按钮（常见选择器：`.close`, `[aria-label="Close"]`, `button:contains("关闭")` 等）
- **授权处理**：尝试查找并点击确认按钮（常见选择器：`button:contains("确认")`, `button:contains("Approve")` 等）
- **页面刷新**：如果检测到页面异常，尝试刷新页面后重试（最多 1 次）

**跳过条件**：
- 重试次数达到上限仍失败
- 页面状态明显异常且无法自动恢复
- 元素定位失败且已尝试多种方式

## 6）性能指标采集实现方式

**✅ 全自动执行，无需手动操作**。性能采集通过以下方式自动完成：

### 6.1 自动注入性能采集脚本（必须在页面加载前注入）

**核心原则**：脚本必须在页面加载前完成注入，确保能够捕获完整的性能数据和 WebSocket 连接。

**注入流程**（严格按照以下顺序执行）：

1. **连接浏览器后立即注册脚本**（在导航到 URL 之前）：
   - 使用 Chrome DevTools Protocol 的 `Page.addScriptToEvaluateOnNewDocument` API
   - 先读取脚本文件内容（`docs/scripts/inject/performance-collector.js`）
   - 调用 `Page.addScriptToEvaluateOnNewDocument` 注册性能采集脚本
   - 如果测试需要 WS 监听，同时注册 WebSocket 监听脚本（`docs/scripts/inject/websocket-monitor.js`）
   - **关键**：必须在导航到目标 URL 之前完成脚本注册

2. **处理已加载页面的情况**：
   - 如果当前页面已加载，先执行页面刷新（`browser_navigate` 的 reload）
   - 刷新前确保脚本已通过 `Page.addScriptToEvaluateOnNewDocument` 注册
   - 刷新后脚本会自动在页面加载前执行

3. **导航到目标 URL**：
   - 使用 `browser_navigate` 导航到目标 URL
   - 已注册的脚本会在页面加载前自动执行

4. **验证脚本注入成功**：
   - 等待页面加载完成（使用 `browser_wait_for`）
   - 检查 `window.__performanceMetrics` 是否存在
   - 如果测试需要 WS 监听，检查 `window.__wsMonitor` 是否存在
   - 如果验证失败，触发自动刷新重试机制（见 6.4 节）

**技术实现要点**：
- 使用 `Page.addScriptToEvaluateOnNewDocument` 确保脚本在页面加载前执行
- 脚本使用浏览器原生 `Performance API` 和 `PerformanceObserver` 获取所有指标
- **用户无需手动执行任何操作**，系统自动完成所有注入流程

### 6.2 采集方式
- **Core Web Vitals**：通过 `PerformanceObserver` 自动监听 LCP、FID、CLS
- **加载性能**：通过 `performance.timing` 和 `performance.getEntriesByType('paint')` 获取 FCP、TTI、TBT
- **资源性能**：通过 `performance.getEntriesByType('resource')` 获取资源加载时间和大小
- **网络性能**：通过 `browser_network_requests` MCP 工具自动获取
- **运行时性能**：通过 `performance.memory` API 自动获取内存使用情况

### 6.3 脚本位置
- 性能采集脚本位于：`docs/scripts/inject/performance-collector.js`
- WebSocket 监听脚本位于：`docs/scripts/inject/websocket-monitor.js`（按需注入）
- 系统在执行 `smoke <URL>` 时会自动读取并注入性能采集脚本
- **WebSocket 监听脚本**：仅在测试步骤涉及图表数据时才注入（见第 7 节）
- **重要**：脚本必须在页面加载前注入，以确保能够获取完整的性能数据
- **用户无需关心脚本内容或手动执行**

### 6.4 脚本注入失败处理（自动刷新重试机制）

**检测时机**：页面加载完成后立即检测脚本是否成功注入。

**检测条件**：
- **性能脚本检测**：检查 `window.__performanceMetrics` 是否存在
  - 如果不存在，说明脚本注入失败
  - 如果存在但返回的 LCP/FID 为 null，说明注入时机过晚（数据采集不完整）
- **WS 脚本检测**（仅在需要时）：检查 `window.__wsMonitor` 是否存在
  - 如果不存在，说明脚本注入失败
  - 如果存在但 `getStats().totalConnections === 0`，可能是正常情况（页面未建立 WS 连接）

**自动刷新重试流程**（完全自动化，无需用户干预）：

1. **检测到注入失败**：
   - 性能脚本不存在或数据不完整
   - 记录失败原因和当前重试次数

2. **自动刷新并重试**（最多重试 2 次）：
   ```
   重试流程（自动执行）：
   
   For 重试次数 in [1, 2]:
     1. 确保脚本已通过 Page.addScriptToEvaluateOnNewDocument 注册
        - 如果未注册，重新注册脚本
     2. 执行页面刷新（browser_navigate 的 reload）
     3. 等待页面加载完成（browser_wait_for）
     4. 重新验证脚本注入是否成功
        - 检查 window.__performanceMetrics 是否存在
        - 如果测试需要 WS，检查 window.__wsMonitor 是否存在
     5. 如果验证成功，继续执行测试流程
     6. 如果验证失败且未达到最大重试次数，继续重试
     7. 如果达到最大重试次数（2 次）仍失败，提示用户手动介入
   ```

3. **重试成功**：
   - 继续执行正常的测试流程
   - 不输出任何提示信息（静默处理）

4. **重试失败（达到最大重试次数）**：
   - 输出错误提示：
     ```
     ⚠️ [脚本注入失败] 已重试 2 次，脚本仍未能成功注入
     
     可能原因：
     - Chrome DevTools Protocol 连接异常
     - 页面加载速度过快，脚本注册时机过晚
     - 浏览器安全策略阻止脚本注入
     
     建议操作：
     1. 检查 Chrome 浏览器是否正常运行
     2. 检查 Chrome DevTools Protocol 连接是否正常（端口 9222）
     3. 手动刷新页面后重试
     ```
   - 暂停执行，等待用户手动介入

**重要说明**：
- **完全自动化**：重试过程完全自动化，无需用户手动操作
- **静默处理**：如果重试成功，不输出任何提示信息，继续正常执行
- **最大重试次数**：2 次，避免无限重试
- **失败后提示**：仅在达到最大重试次数仍失败时，才提示用户手动介入

## 7）WebSocket 监听能力（按需注入）

**✅ 支持监听 WebSocket 连接和消息**。通过自动注入 WebSocket 监听脚本实现：

### 7.1 按需调用原则（重要）

**核心原则**：WebSocket 监听是**按需引用**的，**不要默认在所有模块测试中请求 WS 数据**。

**调用条件**：**只有涉及到图表数据相关的情况才会调用**。

**判断标准**：
- ✅ **需要调用 WS**：测试用例中明确涉及图表数据相关的验证（如 K线图、实时价格图表、BBO 价格图表等）
- ❌ **不需要调用 WS**：
  - 仅测试列表展示、表单提交、按钮点击等非图表相关功能
  - 即使模块是 Perps/Market，如果测试内容不涉及图表数据，也不调用 WS
  - 一体化测试（smoke）中，除非测试步骤明确涉及图表数据，否则不注入 WS 脚本


### 7.2 判断逻辑

**判断依据**：
1. **用例文件内容分析**：分析用例文件中的测试步骤，判断是否涉及图表数据验证
2. **用户指令分析**：分析用户指令，判断是否涉及图表数据相关操作
3. **测试场景分析**：分析测试场景，判断是否需要验证实时数据更新

**示例**：
- ✅ **需要调用**：
  - 测试用例中包含"验证 K线图实时更新"
  - 测试用例中包含"验证 BBO 价格实时更新"
  - 测试用例中包含"验证图表数据准确性"
- ❌ **不需要调用**：
  - 仅测试"Token 收藏/取消收藏"功能
  - 仅测试"表单提交"功能
  - 仅测试"按钮点击"功能
  - 即使模块是 Perps/Market，但测试内容不涉及图表数据

### 7.3 自动拦截 WebSocket（仅在需要时）
- **仅在需要时**注入 `docs/scripts/inject/websocket-monitor.js` 脚本
- 脚本会拦截原生 `WebSocket` 构造函数，监听所有 WebSocket 连接
- **无需手动操作，自动监听所有 WS 连接**（仅在需要时）

### 7.4 监听内容
- **连接状态**：连接打开、关闭、错误、重连
- **消息内容**：发送和接收的所有消息（自动解析 JSON）
- **消息统计**：消息数量、大小、时间戳
- **Perps BBO 数据**：自动提取 buy1/sell1 价格数据

## 8）执行顺序规范

**严格按照以下顺序执行**：

### 8.1 连接浏览器

**直接连接到已运行的 Chrome**（端口 9222）：
- 使用 `mcp_chrome-devtools-9222_list_pages` 获取页面列表，验证连接是否成功
- 使用 `mcp_chrome-devtools-9222_select_page` 选择目标页面
- **无需启动新实例**：直接使用用户当前的 Chrome 窗口

**连接失败处理**：
- 如果连接失败，提示用户运行 `./docs/scripts/start-mcp-chrome.sh` 检查 CDP 状态
- 或提示用户关闭 Chrome 后用调试模式重启：`pkill -f "Google Chrome" && /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222 &`

### 8.2 注册脚本（在导航到 URL 之前）

- **关键步骤**：必须在导航到目标 URL 之前完成脚本注册
- 读取性能采集脚本文件（`docs/scripts/inject/performance-collector.js`）
- 使用 `mcp_chrome-devtools-9222_evaluate_script` 注册性能采集脚本
- **WebSocket 监听脚本**（按需注册）：见第 7 节

### 8.3 导航与验证

1. **导航到目标 URL**：使用 `mcp_chrome-devtools-9222_navigate_page` 导航
2. **等待页面加载完成**：使用 `mcp_chrome-devtools-9222_wait_for` 等待关键元素出现
3. **验证脚本注入成功**：检查 `window.__performanceMetrics` 是否存在
4. **如果验证失败**：自动刷新页面并重试（最多 2 次）

### 8.4 初始状态检查（Check State）

- 使用 `mcp_chrome-devtools-9222_take_snapshot` 获取页面快照
- 分析页面状态：关键元素是否存在、页面是否加载完成
- 检查钱包连接状态：是否已连接、是否有数据
- 检查是否有弹窗遮挡：钱包插件弹窗、授权请求等
- 自动处理常见弹窗（不暂停执行）

### 8.5 采集基础性能指标

- 自动获取网络请求（`mcp_chrome-devtools-9222_list_network_requests`）
- 获取控制台消息（`mcp_chrome-devtools-9222_list_console_messages`）
- 获取 Performance API 数据（脚本已在页面加载前注入）

### 8.6 执行冒烟测试流程（Action + Check State 循环）

- **简要说明测试内容**：在执行前简要说明正在测试的内容
- **显示执行进度**：使用进度表示（如 `1/6` 或 `步骤 1/6`）
- **每个操作前**：使用 `take_snapshot` 检查状态
- **执行操作**：使用 `click`、`fill`、`hover` 等工具
- **每个操作后**：使用 `take_snapshot` 验证结果
- **错误处理**（全自动，不暂停）：遇到阻碍时跳过并继续

### 8.7 集中处理异常步骤

- 汇总所有跳过的步骤
- 尝试重新执行跳过的步骤
- 生成异常处理报告

### 8.8 采集性能指标并生成报告

- 采集交互性能指标（FID、CLS 等）
- 采集最终性能数据
- 生成一体化报告

## 9）用户操作说明

### 9.1 准备工作

**确保 Chrome 带调试端口运行**：
```bash
# 检查 CDP 是否可用
./docs/scripts/start-mcp-chrome.sh

# 如果提示未就绪，关闭 Chrome 后重启
pkill -f "Google Chrome"
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222 &
```

### 9.2 执行模式

- **直接连接**：MCP 直接连接到你当前使用的 Chrome 窗口（推荐）
- **全自动执行**：执行过程中无需人工确认，系统自动完成所有操作和验证
- **错误处理**：遇到阻碍时自动跳过并继续执行，最后集中处理异常步骤
- **透明化采集**：性能采集在后台静默进行，不影响功能验证流程

### 9.3 注意事项

- **无需复制数据**：直接使用当前浏览器的所有状态（钱包、登录、扩展等）
- **无需手动执行脚本**：所有性能采集脚本由系统自动注入和执行
- **只需提供 URL**：系统会自动完成所有性能采集和功能验证
