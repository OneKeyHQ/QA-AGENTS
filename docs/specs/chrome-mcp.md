# Chrome MCP 工具使用规范

> 本节定义使用 Chrome DevTools MCP 进行自动化测试的工具规范。

---

## 1）连接浏览器（调试模式）

- **使用 Chrome DevTools MCP**：连接到本地已打开的 Chrome 浏览器（端口 9222）
- **确保浏览器已启动**：执行冒烟测试前，确保已通过 `docs/scripts/start-mcp-chrome.sh` 启动 Chrome 并开启远程调试
- **使用有数据的钱包**：确保连接的浏览器中已登录钱包且有测试数据

---

## 2）必须使用的工具

- `browser_navigate`：导航到目标 URL
- `browser_wait_for`：等待页面加载完成或关键元素出现
- `browser_snapshot`：**核心工具**，用于状态检查
  - 操作前检查：获取页面快照，分析关键元素是否存在、页面是否可操作
  - 操作后验证：验证操作是否成功、状态是否符合预期
  - 人工干预检测：识别钱包插件弹窗、授权请求、密码输入等需要人工操作的场景
- `browser_network_requests`：获取网络请求数据（包括 WebSocket 连接信息）
- `browser_console_messages`：获取控制台消息和性能指标

---

## 3）可选使用的工具（冒烟测试）

- `browser_click`：点击元素（使用前必须通过 `browser_snapshot` 确认元素存在且可点击）
- `browser_type`：输入文本（使用前必须通过 `browser_snapshot` 确认输入框存在且可编辑）
- `browser_select_option`：选择下拉选项（使用前必须通过 `browser_snapshot` 确认下拉框存在）
- `browser_hover`：悬停元素（使用前必须通过 `browser_snapshot` 确认元素存在）
- `browser_take_screenshot`：截图保存（用于报告或调试）

---

## 4）状态检查工作流（Check State → Action → Check State）

1. **操作前**：使用 `browser_snapshot` 获取页面快照
   - 分析快照中的元素，确认目标元素是否存在
   - 检查页面状态（是否加载完成、是否有错误）
   - 检查是否有弹窗遮挡（钱包插件弹窗、授权请求等）
   - 如检测到需要人工操作，暂停并提示用户
2. **执行操作**：确认状态无误后，执行 UI 操作（`browser_click`、`browser_type` 等）
3. **操作后**：使用 `browser_snapshot` 再次获取页面快照
   - 验证操作是否成功（元素状态变化、页面跳转等）
   - 检查是否有错误提示或异常状态
   - 确认页面状态符合预期

---

## 5）人工干预检测工作流（Human Intervention）

1. **检测触发**：使用 `browser_snapshot` 获取页面快照后，分析快照中的元素
2. **识别需要人工操作的场景**（仅在明显阻碍时提示）：
   - **明显阻碍场景**（需要暂停并提示）：
     - 元素找不到且重试失败（已尝试多种定位方式仍无法找到）
     - 操作失败且无法自动恢复（点击无响应、输入失败等，且已重试失败）
     - 页面异常导致无法继续（错误提示、加载失败等，且无法自动恢复）
     - 任何导致测试流程完全无法继续的问题
   - **自动处理场景**（不暂停执行）：
     - 钱包插件弹窗：尝试自动关闭或跳过，不暂停
     - 授权请求：尝试自动处理或跳过，不暂停
     - 密码输入：尝试自动填充或跳过，不暂停
     - 常见弹窗：尝试自动关闭或跳过，不暂停
     - 元素暂时不可见：等待后重试，不暂停
     - 操作暂时失败：重试后继续，不暂停
3. **暂停并提示**：仅在检测到明显阻碍时，暂停执行并输出提示信息
   ```
   ⚠️ [需要人工操作] <具体操作内容>
   
   例如：
   - 元素"最优价"按钮找不到，已尝试多种定位方式仍无法找到，请检查页面状态
   - 点击操作失败，已重试3次仍无响应，请检查页面是否正常
   - 页面出现错误提示，无法自动恢复，请手动处理
   
   完成后请回复"继续"或"ok"
   ```
4. **等待用户确认**：用户完成手动操作后，回复"继续"或"ok"
5. **继续前检查**：收到用户确认后，再次使用 `browser_snapshot` 检查状态，确认操作已完成
6. **继续执行**：确认状态正确后，继续执行下一步操作

## 6）性能指标采集实现方式

**✅ 全自动执行，无需手动操作**。性能采集通过以下方式自动完成：

### 6.1 自动注入性能采集脚本（必须在页面加载前注入）

**核心原则**：脚本必须在页面加载前完成注入，确保能够捕获完整的性能数据和 WebSocket 连接。

**注入流程**（严格按照以下顺序执行）：

1. **连接浏览器后立即注册脚本**（在导航到 URL 之前）：
   - 使用 Chrome DevTools Protocol 的 `Page.addScriptToEvaluateOnNewDocument` API
   - 先读取脚本文件内容（`docs/scripts/inject/performance-collector.js`）
   - 调用 `Page.addScriptToEvaluateOnNewDocument` 注册性能采集脚本
   - 如果测试需要 WS 监听，同时注册 WebSocket 监听脚本（`docs/scripts/inject/websocket-monitor.js`）
   - **关键**：必须在导航到目标 URL 之前完成脚本注册

2. **处理已加载页面的情况**：
   - 如果当前页面已加载，先执行页面刷新（`browser_navigate` 的 reload）
   - 刷新前确保脚本已通过 `Page.addScriptToEvaluateOnNewDocument` 注册
   - 刷新后脚本会自动在页面加载前执行

3. **导航到目标 URL**：
   - 使用 `browser_navigate` 导航到目标 URL
   - 已注册的脚本会在页面加载前自动执行

4. **验证脚本注入成功**：
   - 等待页面加载完成（使用 `browser_wait_for`）
   - 检查 `window.__performanceMetrics` 是否存在
   - 如果测试需要 WS 监听，检查 `window.__wsMonitor` 是否存在
   - 如果验证失败，触发自动刷新重试机制（见 6.4 节）

**技术实现要点**：
- 使用 `Page.addScriptToEvaluateOnNewDocument` 确保脚本在页面加载前执行
- 脚本使用浏览器原生 `Performance API` 和 `PerformanceObserver` 获取所有指标
- **用户无需手动执行任何操作**，系统自动完成所有注入流程

### 6.2 采集方式
- **Core Web Vitals**：通过 `PerformanceObserver` 自动监听 LCP、FID、CLS
- **加载性能**：通过 `performance.timing` 和 `performance.getEntriesByType('paint')` 获取 FCP、TTI、TBT
- **资源性能**：通过 `performance.getEntriesByType('resource')` 获取资源加载时间和大小
- **网络性能**：通过 `browser_network_requests` MCP 工具自动获取
- **运行时性能**：通过 `performance.memory` API 自动获取内存使用情况

### 6.3 脚本位置
- 性能采集脚本位于：`docs/scripts/inject/performance-collector.js`
- WebSocket 监听脚本位于：`docs/scripts/inject/websocket-monitor.js`（按需注入）
- 系统在执行 `smoke <URL>` 时会自动读取并注入性能采集脚本
- **WebSocket 监听脚本**：仅在测试步骤涉及图表数据时才注入（见第 7 节）
- **重要**：脚本必须在页面加载前注入，以确保能够获取完整的性能数据
- **用户无需关心脚本内容或手动执行**

### 6.4 脚本注入失败处理（自动刷新重试机制）

**检测时机**：页面加载完成后立即检测脚本是否成功注入。

**检测条件**：
- **性能脚本检测**：检查 `window.__performanceMetrics` 是否存在
  - 如果不存在，说明脚本注入失败
  - 如果存在但返回的 LCP/FID 为 null，说明注入时机过晚（数据采集不完整）
- **WS 脚本检测**（仅在需要时）：检查 `window.__wsMonitor` 是否存在
  - 如果不存在，说明脚本注入失败
  - 如果存在但 `getStats().totalConnections === 0`，可能是正常情况（页面未建立 WS 连接）

**自动刷新重试流程**（完全自动化，无需用户干预）：

1. **检测到注入失败**：
   - 性能脚本不存在或数据不完整
   - 记录失败原因和当前重试次数

2. **自动刷新并重试**（最多重试 2 次）：
   ```
   重试流程（自动执行）：
   
   For 重试次数 in [1, 2]:
     1. 确保脚本已通过 Page.addScriptToEvaluateOnNewDocument 注册
        - 如果未注册，重新注册脚本
     2. 执行页面刷新（browser_navigate 的 reload）
     3. 等待页面加载完成（browser_wait_for）
     4. 重新验证脚本注入是否成功
        - 检查 window.__performanceMetrics 是否存在
        - 如果测试需要 WS，检查 window.__wsMonitor 是否存在
     5. 如果验证成功，继续执行测试流程
     6. 如果验证失败且未达到最大重试次数，继续重试
     7. 如果达到最大重试次数（2 次）仍失败，提示用户手动介入
   ```

3. **重试成功**：
   - 继续执行正常的测试流程
   - 不输出任何提示信息（静默处理）

4. **重试失败（达到最大重试次数）**：
   - 输出错误提示：
     ```
     ⚠️ [脚本注入失败] 已重试 2 次，脚本仍未能成功注入
     
     可能原因：
     - Chrome DevTools Protocol 连接异常
     - 页面加载速度过快，脚本注册时机过晚
     - 浏览器安全策略阻止脚本注入
     
     建议操作：
     1. 检查 Chrome 浏览器是否正常运行
     2. 检查 Chrome DevTools Protocol 连接是否正常（端口 9222）
     3. 手动刷新页面后重试
     ```
   - 暂停执行，等待用户手动介入

**重要说明**：
- **完全自动化**：重试过程完全自动化，无需用户手动操作
- **静默处理**：如果重试成功，不输出任何提示信息，继续正常执行
- **最大重试次数**：2 次，避免无限重试
- **失败后提示**：仅在达到最大重试次数仍失败时，才提示用户手动介入

## 7）WebSocket 监听能力（按需注入）

**✅ 支持监听 WebSocket 连接和消息**。通过自动注入 WebSocket 监听脚本实现：

### 7.1 按需调用原则（重要）

**核心原则**：WebSocket 监听是**按需引用**的，**不要默认在所有模块测试中请求 WS 数据**。

**调用条件**：**只有涉及到图表数据相关的情况才会调用**。

**判断标准**：
- ✅ **需要调用 WS**：测试用例中明确涉及图表数据相关的验证（如 K线图、实时价格图表、BBO 价格图表等）
- ❌ **不需要调用 WS**：
  - 仅测试列表展示、表单提交、按钮点击等非图表相关功能
  - 即使模块是 Perps/Market，如果测试内容不涉及图表数据，也不调用 WS
  - 一体化测试（smoke）中，除非测试步骤明确涉及图表数据，否则不注入 WS 脚本


### 7.2 判断逻辑

**判断依据**：
1. **用例文件内容分析**：分析用例文件中的测试步骤，判断是否涉及图表数据验证
2. **用户指令分析**：分析用户指令，判断是否涉及图表数据相关操作
3. **测试场景分析**：分析测试场景，判断是否需要验证实时数据更新

**示例**：
- ✅ **需要调用**：
  - 测试用例中包含"验证 K线图实时更新"
  - 测试用例中包含"验证 BBO 价格实时更新"
  - 测试用例中包含"验证图表数据准确性"
- ❌ **不需要调用**：
  - 仅测试"Token 收藏/取消收藏"功能
  - 仅测试"表单提交"功能
  - 仅测试"按钮点击"功能
  - 即使模块是 Perps/Market，但测试内容不涉及图表数据

### 7.3 自动拦截 WebSocket（仅在需要时）
- **仅在需要时**注入 `docs/scripts/inject/websocket-monitor.js` 脚本
- 脚本会拦截原生 `WebSocket` 构造函数，监听所有 WebSocket 连接
- **无需手动操作，自动监听所有 WS 连接**（仅在需要时）

### 7.4 监听内容
- **连接状态**：连接打开、关闭、错误、重连
- **消息内容**：发送和接收的所有消息（自动解析 JSON）
- **消息统计**：消息数量、大小、时间戳
- **Perps BBO 数据**：自动提取 buy1/sell1 价格数据

### 7.5 Perps BBO 测试支持
- 自动监听 Hyperliquid BBO WebSocket 订阅
- 提取 buy1/sell1 价格数据
- 检测 WS 断连/重连事件
- 验证 BBO 更新节流（仅价格变化触发更新）
- 验证 BBO 数据异常处理

### 7.6 获取 WebSocket 数据
- 执行 `window.__wsMonitor.getReport()` 获取完整报告
- 执行 `window.__wsMonitor.getBBOData()` 获取 BBO 数据
- 执行 `window.__wsMonitor.getConnections()` 获取连接信息
- 系统会在冒烟测试报告中自动包含 WebSocket 数据（仅当注入了 WS 脚本时）

## 8）执行顺序规范

**严格按照以下顺序执行，确保脚本在页面加载前成功注入**：

1. **连接浏览器**：
   - 使用 Chrome DevTools MCP 连接到本地已打开的 Chrome 浏览器（端口 9222）
   - 验证连接是否成功

2. **注册脚本（在导航到 URL 之前）**：
   - **关键步骤**：必须在导航到目标 URL 之前完成脚本注册
   - 读取性能采集脚本文件（`docs/scripts/inject/performance-collector.js`）
   - 使用 Chrome DevTools Protocol 的 `Page.addScriptToEvaluateOnNewDocument` API 注册性能采集脚本
   - **WebSocket 监听脚本**（按需注册）：
     - 分析测试用例内容，判断是否需要 WS 监听（见第 7 节）
     - 如果需要，读取 WebSocket 监听脚本文件（`docs/scripts/inject/websocket-monitor.js`）
     - 使用 `Page.addScriptToEvaluateOnNewDocument` 注册 WS 监听脚本
   - **验证脚本注册成功**：确认 API 调用返回成功

3. **处理已加载页面的情况**：
   - 检查当前页面是否已加载
   - 如果已加载，执行页面刷新（`browser_navigate` 的 reload）
   - 刷新后，已注册的脚本会在页面加载前自动执行

4. **导航到目标 URL**：
   - 使用 `browser_navigate` 导航到目标 URL
   - 已注册的脚本会在页面加载前自动执行

5. **等待页面加载完成**：
   - 使用 `browser_wait_for` 等待页面加载完成
   - 等待关键元素出现或页面稳定

6. **验证脚本注入成功**（自动执行）：
   - 检查 `window.__performanceMetrics` 是否存在
   - 如果测试需要 WS 监听，检查 `window.__wsMonitor` 是否存在
   - **如果验证失败，自动触发刷新重试机制**（见 6.4 节）：
     - 自动刷新页面并重试（最多 2 次）
     - 如果重试成功，继续执行
     - 如果重试失败，提示用户手动介入
4. **初始状态检查（Check State）**：
   - 使用 `browser_snapshot` 获取页面快照
   - 分析页面状态：关键元素是否存在、页面是否加载完成
   - 检查钱包连接状态：是否已连接、是否有数据
   - 检查是否有弹窗遮挡：钱包插件弹窗、授权请求等
   - 如检测到需要人工操作，暂停并提示用户
5. **采集基础性能指标**：自动获取网络请求、控制台消息、Performance API 数据（脚本已在页面加载前注入）
6. **执行冒烟测试流程（Action + Check State 循环）**：
   - **简要说明测试内容**：在执行前简要说明正在测试的内容（如"正在测试限价单最优价格BBO功能"）
   - **显示执行进度**：使用进度表示（如 `1/6` 或 `步骤 1/6`）显示当前执行进度
   - **每个操作前**：使用 `browser_snapshot` 检查状态（元素是否存在、页面是否可操作、是否有弹窗）
   - **执行操作**：如提供用例文件，**执行用例文件中的所有步骤**（不限于 P0，包括所有优先级）
   - **每个操作后**：使用 `browser_snapshot` 验证结果（操作是否成功、状态是否符合预期）
   - **检测人工干预**（仅在明显阻碍时提示）：
     - 自动处理常见场景（钱包插件弹窗、授权请求、密码输入等），不暂停执行
     - **仅在明显阻碍时暂停**：元素找不到且重试失败、操作失败且无法自动恢复、页面异常导致无法继续等
   - **等待用户确认**：仅在明显阻碍时，用户完成手动操作后，回复"继续"或"ok"，Agent 再次检查状态后继续
7. **采集交互性能指标**：在执行功能验证过程中，自动采集 FID、CLS 等交互指标
8. **再次采集运行时性能指标**：功能验证完成后，自动采集最终性能数据
9. **生成完整报告**：汇总所有指标并输出一体化报告

---

## 9）用户操作说明

- **调试模式**：冒烟测试连接本地已打开的 Chrome 浏览器，确保钱包已登录且有数据
- **人工干预**：当检测到需要手动操作时（如钱包插件弹窗、授权确认等），Agent 会暂停并提示，用户完成操作后回复"继续"即可
- **无需手动执行任何脚本**：所有性能采集脚本由系统自动注入和执行
- **无需关心实现细节**：只需提供 URL 和用例文件（可选），系统会自动完成所有性能采集
- **透明化采集**：性能采集在后台静默进行，不影响功能验证流程
