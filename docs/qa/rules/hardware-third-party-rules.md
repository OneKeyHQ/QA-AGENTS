# 硬件-第三方 测试规则文档

> 本文档记录第三方硬件设备在 OneKey Account/Hardware 场景下的专项测试规则。  
> 生成第三方硬件相关测试用例时，必须参考本文档。  
> 当前已沉淀：Ledger、Trezor。
> 关联需求：`docs/qa/requirements/Account-Ledger设备识别与匹配规则.md`、`docs/qa/requirements/Hardware-Ledger安装APP.md`、`docs/qa/requirements/Account-Trezor连接与设备交互.md`

---

## 1. 适用范围

| 维度 | 范围 |
|-----|------|
| 设备大类 | Ledger、Trezor（仅 Trezor 无 App 安装逻辑） |
| 功能域 | 设备连接、Wallet 添加、Account 添加、设备匹配校验；其中 App 安装能力仅适用于 Ledger |
| 连接方式 | USB、BLE（按设备能力生效） |
| 端 | Desktop（后续端能力扩展时同步补充） |
| 链路优先级 | BTC、EVM 为 P0 回归链路；SOL、TRON 为安装与交互回归链路 |

## 2. Ledger（已沉淀）

> 连接支持网络：BTC / EVM / SOL / TRON。  
> 安装能力支持网络：BTC / ETH / TRX / SOL。  
> 安装能力命名口径按 App 名称使用 `ETH / TRX`；账户交互回归按网络类别使用 `EVM / TRON`。

### 2.1 USB 设备识别、连接与账户规则

#### 2.1.1 单设备连接原则

1. USB 链路下同一时间仅允许一台 Ledger 处于可连接/已连接状态。  
2. 系统识别到 2 台及以上 USB 设备并存时，拦截连接流程并提示用户。  
3. 当前版本不提供“手动切换/指定目标设备”的交互入口。

#### 2.1.2 多设备行为约束

| 场景 | 期望行为 |
|-----|---------|
| USB 仅接入 1 台设备 | 正常进入连接流程，可完成 wallet/account 业务 |
| USB 同时接入 ≥2 台设备 | 弹出提示「仅允许同时连接一台设备」（或等价文案），不建立连接 |
| 多设备提示后断开至 1 台 | 提示消失，可重新发起连接并完成业务 |
| 连接过程中再插入第二台 | 触发多设备提示，不允许在双设备并存下继续建立新连接 |
| 会话中设备变化 | 需重新触发识别；从多台变为单台后，方可继续连接 |

#### 2.1.3 Wallet 维度

| 场景 | 规则 |
|-----|------|
| 同一设备重复添加 wallet | 允许；每次添加可生成新 wallet |
| 设备唯一性 | USB 侧无稳定设备唯一 ID，不以“设备实例”做强限制 |

#### 2.1.4 Account 维度（核心）

1. 校验锚点：以“网络首地址 Hash”作为该网络后续添加的一致性锚点。  
2. 命中锚点网络：再次添加同网络账户时必须校验首地址 Hash 一致性。  
3. 不命中锚点网络：若该网络尚无已添加地址，不做助记词一致性限制。

#### 2.1.5 设备不匹配判定

| 条件 | 结果 |
|-----|------|
| 同网络首地址 Hash 一致 | 允许继续添加 |
| 同网络首地址 Hash 不一致 | 拦截并提示“设备不匹配” |
| 同助记词不同 Ledger 设备 | 允许继续（校验通过） |
| 不同助记词设备且网络已有锚点 | 拦截并提示“设备不匹配” |

### 2.2 BLE 规则

#### 2.2.1 设备标识与聚合

| 场景 | 规则 |
|-----|------|
| 识别设备 | 使用蓝牙标识进行设备识别 |
| 添加 wallet 时蓝牙标识可匹配 | 进入地址匹配判断流程 |
| 添加 wallet 时蓝牙标识不可匹配 | 创建新 wallet |

#### 2.2.2 匹配分支

| 条件 | 结果 |
|-----|------|
| 蓝牙可匹配 且 任一网络地址可匹配 | 合并到已有 wallet |
| 蓝牙可匹配 但 全部网络地址不可匹配 | 创建新 wallet |

### 2.3 Ledger App 安装规则

#### 2.3.1 适用范围

| 项目 | 规则 |
|-----|------|
| 支持安装的网络 | BTC / ETH / TRX / SOL |
| 批量安装场景 | Onboarding、网络选择投资组合（旧 all networks）创建账户 |
| 单网络安装场景 | 创建地址、核对地址、签名 |

#### 2.3.2 Onboarding / 投资组合批量安装触发条件

| 条件 | 结果 |
|-----|------|
| 4 个支持网络 App 全部未安装 | 弹出复选框，让用户选择需要安装的 App |
| 4 个支持网络中至少有 1 个可用 App | 不进入批量安装流程 |

#### 2.3.3 批量安装执行规则

| 场景 | 规则 |
|-----|------|
| 用户勾选多个待安装网络 | 按产品定义顺序逐个安装，不并行安装 |
| 安装完成后继续创建 | 仅对已成功安装并可用的网络继续创建地址 |
| 某网络未勾选安装 | 该网络不创建地址 |
| 存在已安装 App 的设备 | 直接按已安装 App 对应网络继续交互，不弹复选框 |

#### 2.3.4 单网络按需安装规则

| 场景 | 规则 |
|-----|------|
| 创建地址时目标网络 App 未安装 | 提示安装对应 App，安装完成后继续创建地址 |
| 核对地址时目标网络 App 未安装 | 提示安装对应 App，安装完成后继续核对地址 |
| 签名时目标网络 App 未安装 | 提示安装对应 App，安装完成后继续签名 |
| 当前网络 App 已安装 | 直接进入原交互流程 |

#### 2.3.5 投资组合创建地址按单网络交互处理

| 项目 | 规则 |
|-----|------|
| 执行模型 | 投资组合勾选网络创建地址，按连续单网络交互处理 |
| 统一结果页 | 不采用“一个结果页统一处理所有未安装网络”的模式 |
| 网络处理顺序 | 对每个勾选网络，按单网络交互顺序逐个处理是否需要安装 App |
| 已安装网络 | 逐个进入对应网络的创建地址流程 |
| 未安装且未补装成功网络 | 不创建地址，不纳入成功创建结果 |
| 前序成功网络 | 不因后续网络未安装或安装失败而回滚 |

### 2.4 异常提示规则

| 异常类型 | 规则 |
|---------|------|
| 断网 | 必须提示网络异常 / 安装失败，不可误提示为设备问题 |
| 设备空间不足 | 必须提示设备空间不足，不可继续安装后续目标 App |
| 设备断线 | 必须提示设备断开或连接中断，当前安装与交互停止 |

### 2.5 账户交互验证（四链）

#### 2.5.1 验证最小集合（P0）

| 类别 | 用例要求 |
|-----|---------|
| BTC | 创建地址、签名、转账流程可达 |
| EVM | 创建地址、签名、转账流程可达 |
| SOL | 创建地址、签名、转账流程可达 |
| TRON | 创建地址、签名、转账流程可达 |
| 不匹配分支 | 在“网络有锚点”前提下可稳定触发“设备不匹配” |
| 安装分支 | 在“全未安装”“部分已安装”“单网络未安装”三类状态下行为正确 |

### 2.6 测试设计强制点

1. 必测“网络已建立锚点”和“网络未建立锚点”两类前置状态。  
2. 必测“同助记词跨设备”与“不同助记词跨设备”两类对照样本。  
3. 必测多设备并存时的「仅允许同时连接一台设备」提示与连接拦截。  
4. 必测 BLE 聚合与新建 wallet 两条分支。  
5. 必测“4 个 App 全未安装才弹复选框”与“存在任一可用 App 时不弹复选框”两类安装入口判定。  
6. 必测投资组合勾选多个网络时，按连续单网络交互处理的执行模型。  
7. 必测断网、设备空间不足、设备断线三类异常提示。  
8. 错误提示必须验证文案语义与触发时机，至少覆盖“设备不匹配”“网络异常”“空间不足”“设备断线”四类。

### 2.7 风险清单

- 多设备插拔时序导致提示未及时出现/消失，或误放行双设备连接。  
- 锚点网络与新网络切换时，校验上下文串链。  
- BLE 聚合条件被误用为放行条件，造成错误归并。  
- 批量安装与单网络交互状态机串联错误，导致“已安装网络被错误跳过”或“未安装网络被错误创建”。  
- 存在一个可用 App 时误进入批量安装流程。  
- 安装异常原因被统一兜底，用户无法区分断网、空间不足和设备断线。  
- 四链交互（BTC/EVM/SOL/TRON）中任一链的创建地址/签名/转账流程出现兼容性回退。

## 3. Trezor（已沉淀）

> 连接支持网络：BTC / EVM / SOL / TRON。
> Trezor 不存在 Ledger 式链 App 安装逻辑，不生成安装 App 相关场景。
> 当前需求目标覆盖 Safe 7 与 Model One 等旧设备差异；正式用例产出前仍需确认旧设备启用范围和派生阈值包含关系。

### 3.1 设备发现与连接规则

1. Desktop 支持 USB；设备与 Desktop 均具备 BLE 能力时支持蓝牙。
2. USB 与蓝牙分属独立连接页面；USB 页面只搜索和展示 USB 设备，蓝牙页面只搜索和展示 BLE 设备，不同时发现或混合展示两类结果。
3. 当前连接页面允许展示多台同通道的 Trezor，不套用 Ledger USB「仅允许同时连接一台设备」规则。
4. 用户选择目标设备后，只建立该设备的当前交互会话；其他已发现设备不阻塞连接。
5. 同一设备分别经 USB、BLE 页面连接时，仍按 DeviceID 识别为同一设备，不按两个设备创建钱包。
6. 同一设备同一时刻只允许一个客户端交互；设备被其他 App/客户端占用时，提示设备忙并终止当前操作，释放后允许重试。

### 3.2 DeviceID 与 Wallet 规则

| 场景 | 规则 |
|-----|------|
| 设备身份 | 以固件返回的 `device_id` 为核心身份；connectId 只表示 USB/BLE 连接通道 |
| 同一未重置设备重复添加标准钱包 | 定位并复用已存在钱包，刷新设备与传输信息，不创建重复标准钱包 |
| 同一设备重置后添加 | DeviceID 变化，创建新钱包 |
| 重置后恢复相同助记词 | 仍创建新钱包，不因助记词相同覆盖旧钱包 |
| 另一设备使用相同助记词 | 视为另一设备，不与原钱包合并 |
| 隐藏钱包身份 | 同一设备下使用 `passphraseState` 区分标准钱包与各隐藏钱包 |

### 3.3 USB→BLE 绑定规则

1. 仅在设备支持 BLE、Desktop BLE 可用且已有 USB 钱包记录时展示绑定入口。
2. 扫描到 BLE 候选后，必须连接候选设备读取 DeviceID；仅当候选 DeviceID 与 USB 钱包 DeviceID 一致时写入同一设备记录。
3. 候选 DeviceID 不一致、候选要求新配对、原设备已重置或使用同助记词的另一设备，均判定为非目标设备，不得绑定。
4. 单个候选失败后允许继续选择其他候选，不把一次不匹配升级为整个绑定流程失败。
5. 已绑定 BLE connectId 失效时保留重新绑定入口，允许覆盖旧值。

### 3.4 Onboarding、PIN 与安全码规则

| 场景 | 规则 |
|-----|------|
| 未初始化设备 | 返回 `initialized = false` 时提示先初始化，立即中断钱包创建；OneKey 不代替 Trezor 完成初始化 |
| Model One 等旧按键设备 | 使用 App 端 PIN 位置矩阵；App 不展示设备实际数字，PIN 错误/取消/断开均中断当前交互 |
| Safe 7 首次配对 | 用户需在 App 输入设备显示的安全码；空值不可提交，错误/取消/超时不保存配对关系 |
| Safe 7 后续连接 | 复用已保存的 THP 配对凭证；移除设备或设备重置后重新配对 |
| 触屏设备 PIN | PIN 在设备端输入，App 仅提示用户操作设备 |

### 3.5 Passphrase 与隐藏钱包规则

1. 支持使用 Passphrase 创建隐藏钱包；空 Passphrase 等价于标准钱包。
2. 地址派生、签名和钱包指纹必须使用该钱包对应的 `passphraseState`，状态不匹配时要求重新输入。
3. 同一设备输入相同 Passphrase 时定位到已有隐藏钱包。
4. Safe 7 在有效会话内可保持多个隐藏钱包状态，切换已保持钱包时不重复输入 Passphrase。
5. 早期设备同一时间只保持一个隐藏钱包交互状态，切换钱包需重新输入 Passphrase。
6. 不得在 `passphraseState` 失效后继续派生或签名，避免标准钱包与隐藏钱包地址串用。

### 3.6 网络与地址派生限制

| 网络 / 派生模式 | 限制阈值 |
|----------------|---------|
| ETH BIP44 | 1,000,000 |
| TRON | 1,000,000 |
| SOL | 100 |
| ETH Ledger Live | 100 |
| ETH Ledger Legacy | 100 |
| BTC | 当前无业务限制 |

规则：

1. 超过设备/SDK允许范围时终止派生并显示路径或序号不支持错误，不得永久 loading。
2. 当前资料未明确阈值本身是否可用；正式用例必须先确认 `阈值 - 1 / 阈值 / 阈值 + 1`。
3. BTC 无当前业务限制不等于无限制，仍需遵守 BIP32 序号格式和 SDK 参数合法性。
4. 四链最小回归范围为创建地址、核对地址、签名和转账主流程。

### 3.7 设备管理规则

1. Trezor 设备详情展示设备名称、型号、固件版本和连接状态等可用信息。
2. 已初始化设备未手动命名且设备 label 为空时，App 使用设备型号作为展示名称，不显示空名称。
3. 同一 DeviceID 的设备名称在第二个 OneKey App 或 Trezor Suite 中修改后，当前 App 在下一次读取设备信息的交互中（例如核对地址）同步最新名称；关联钱包名称和设备详情不得继续显示旧名称。
4. 设置项按设备型号、features 字段和 SDK 方法能力动态展示，不要求所有型号一致。
5. 自动锁定、亮度、触觉反馈、PIN、Passphrase 等能力仅在设备明确支持时展示；不支持项隐藏或明确不可用。
6. 设备拒绝、断开或方法不支持时，不更新 App 本地状态。
7. 不展示 OneKey 专属设置、固件验真或 Ledger App 管理入口。

### 3.8 异常提示规则

| 异常 | 规则 |
|-----|------|
| 设备忙/被其他客户端占用 | 显示设备忙语义，终止当前操作，释放后允许重试 |
| 设备未初始化 | 显示初始化提示并中断创建 |
| USB/BLE 断线 | 显示设备断开/连接中断，停止当前交互 |
| BLE DeviceID 不匹配 | 当前候选不可绑定，允许继续选择其他候选 |
| 安全码错误/取消/超时 | 不保存配对关系，可重新发起 |
| PIN 错误/不匹配 | 显示 PIN 错误语义，不统一兜底为连接失败 |
| Passphrase 状态不匹配 | 要求重新输入，不继续派生或签名 |
| 派生路径/序号超限 | 显示路径不支持错误并关闭处理态 |

### 3.9 测试设计强制点

1. 必须区分「USB/BLE 独立页面」「同通道多设备列表」与「单设备单客户端独占交互」，不得写成 USB/BLE 结果同时发现，也不得把多设备发现误写为只允许连接一台。
2. 必测同一设备重复添加、重置后相同物理设备、同助记词另一设备三类身份分支。
3. 必测 USB→BLE 的 DeviceID 一致、不一致、要求新配对、旧 BLE 标识失效重绑四类分支。
4. 必测未初始化设备中断创建，且数据库不残留新钱包。
5. 按启用机型覆盖旧设备 App PIN 与 Safe 7 安全码两类交互。
6. 必测标准钱包、多个 Passphrase 隐藏钱包及会话失效后的重新输入。
7. 派生阈值正式确认后，再按边界三点法产出用例；未确认前不得把阈值包含关系写死。
8. 设备设置按型号能力矩阵设计，不复制 OneKey 全量设置清单。
9. Trezor 不生成 Ledger App 安装场景。
10. 必测未手动命名且 label 为空时按设备型号展示名称，不得显示空名称。
11. 必测在第二个 OneKey App 或 Trezor Suite 修改名称后，返回当前 App 并通过核对地址等设备交互同步最新名称；同一 DeviceID 不得因此新增重复钱包。

### 3.10 风险与当前实现差异

- 产品规则要求 USB、蓝牙页面分别只搜索当前通道；`app-monorepo` `x` 分支当前仍存在共享扫描后按页面过滤结果的实现，需要开发对齐。
- `app-monorepo` `x` 分支厂商配置仍注明当前主支持对象为 Safe 7，Model One 正式启用范围需确认。
- 已有旧设备 PIN 矩阵实现结构，但需确认发布构建是否开放对应机型。
- Safe 7 多隐藏钱包会话与旧设备单会话差异需要真机联调确认。
- 2026-07-09 Trezor 修复历史包含：超大 SOL 派生序号触发 `Forbidden key path` 后处理弹窗卡住、PIN mismatch 文案错误、跨端设备名称不一致；后续用例需覆盖这些回归风险。
- 派生阈值是否包含边界值尚未确认，提前写死会导致错误断言。
