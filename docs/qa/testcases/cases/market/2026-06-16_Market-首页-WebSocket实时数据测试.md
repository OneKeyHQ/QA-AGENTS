# Market - 首页 WebSocket 实时数据测试
> 生成时间：2026-06-16
> 规则文档：`docs/qa/rules/market-rules.md`
> 测试端：Desktop / Web / Extension

## 测试目标

验证 Market 首页在切换到 WebSocket 数据源后：

- 能建立并维持 WebSocket 连接
- 能正确订阅首页所需的数据
- 能将 WebSocket 推送正确映射到 UI 列表与热门卡片
- 断网 / 断连后能正确降级、重连、恢复
- 高频更新下 UI 不闪烁、不明显抖动

---

## 接口可见性说明

### 1. 能不能看到 WebSocket 接口

**可以看到运行时接口**，包括：

- `wss://...` 连接地址
- 握手请求 / 响应（含 query、headers、101 状态）
- 客户端发送的订阅消息
- 服务端推送的数据帧
- 心跳 / ping-pong / 重连行为

**通常看不到像 REST 一样的 Apifox / OpenAPI 文档定义**。  
WebSocket 更适合通过 **运行时抓包** 或 **源码定位** 进行验证。

### 2. 推荐观察方式

#### 方式 A：Chrome / Electron DevTools

1. 打开页面
2. 进入 `Network`
3. 过滤 `WS`
4. 观察：
   - 是否建立 `wss://` 连接
   - 握手是否返回 `101 Switching Protocols`
   - `Frames` 中是否有订阅消息和价格推送

#### 方式 B：项目内置 WS 监听脚本

项目已内置：

- `docs/skills/scripts/inject/websocket-monitor.js`
- `docs/skills/specs/smoke-test.md` 中 `ws <URL>` 监听口径

可用于捕获：

- 连接数
- 重连次数
- 最近消息
- 消息大小
- 错误数量

## 测试范围

本轮聚焦首页可见的两类数据区：

- 热门趋势卡片
- 自选 / 现货 / 合约列表

本轮重点字段：

- 价格
- 24h 涨跌幅
- 24h 交易额
- 合约场景补充：资金费率、持仓量（如页面展示）

## 前置条件

1. 可进入 Market 首页
2. 测试环境网络稳定，可主动断网 / 恢复网络
3. 可打开 DevTools 或使用 CDP 抓取 WS
4. 至少选定 1 个稳定观测标的用于比对，如：
   - 现货：BTC / ETH / SOL
   - 合约：BTC / ETH
5. 如使用内置监听脚本，需在页面加载前完成注入

- WS URL

- UI 采样时间

## 用例

### P0 核心链路

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|--------|------|---------|---------|
| ❗️❗️P0❗️❗️ | 首页进入后建立 WS 连接 | 1. 打开 Market 首页<br>2. 在 DevTools `Network -> WS` 观察连接 | 1. 至少建立 1 条与 Market 数据相关的 `wss://` 连接<br>2. 握手返回状态码 `101`<br>3. 不存在持续错误重试风暴 |
| ❗️❗️P0❗️❗️ | 首页进入后发送订阅消息 | 1. 打开 `Frames`<br>2. 观察客户端发送消息 | 1. 能看到订阅类消息（如 subscribe / auth / topic join）<br>2. 订阅内容覆盖当前页面所需列表或热门卡片数据 |
| ❗️❗️P0❗️❗️ | 现货列表价格映射正确 | 1. 保持在「现货」Tab<br>2. 选定 1 个可稳定更新的 Token（如 BTC）<br>3. 同时记录 UI 价格与最近一条 WS 推送值 | 1. UI 价格与 WS 对应字段一致或在产品允许精度范围内一致<br>2. 价格格式正确（USD） |
| ❗️❗️P0❗️❗️ | 现货列表涨跌幅映射正确 | 1. 保持在「现货」Tab<br>2. 观察选定 Token 的 24h 涨跌幅<br>3. 对照 WS 对应字段 | 1. UI 涨跌幅与 WS 数据一致<br>2. 正负号正确<br>3. 颜色正确：涨绿跌红 |
| ❗️❗️P0❗️❗️ | 现货列表交易额映射正确 | 1. 记录选定 Token 的交易额<br>2. 对照 WS 推送中的交易额字段 | 1. UI 与 WS 数值一致或按展示单位正确换算<br>2. 单位格式正确（如 `$5.17B`） |
| ❗️❗️P0❗️❗️ | 合约列表价格 / 涨跌 / 交易额实时更新 | 1. 切到「合约」Tab<br>2. 观察 BTC/ETH 等合约行<br>3. 等待 WS 推送 | 1. 价格、涨跌、交易额可随 WS 推送更新<br>2. 不需要手动刷新页面 |
| ❗️❗️P0❗️❗️ | 热门趋势卡片随 WS 更新 | 1. 停留首页顶部热门卡片区<br>2. 记录任一卡片涨跌幅<br>3. 等待推送并再次观察 | 1. 卡片中的涨跌幅可实时变化<br>2. 更新后与对应 WS 推送一致 |
| ❗️❗️P0❗️❗️ | 更新节流生效 | 1. 持续观察 10-20 秒<br>2. 关注高频更新标的的 UI 表现 | 1. 数值可更新<br>2. 不出现明显闪烁、整行重绘抖动、频繁跳位 |

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|--------|------|---------|---------|
| P1 | 切换主标签后订阅跟随切换 | 1. 依次切换「自选 / 现货 / 合约」<br>2. 观察订阅消息 / 推送类型变化 | 1. 标签切换后订阅集合按页面需求调整<br>2. 不再展示的数据不继续错误刷新 |
| P1 | 切换现货筛选条件后订阅内容更新 | 1. 进入「现货」<br>2. 切换网络或分类筛选<br>3. 观察后续 WS 消息 | 1. 新列表对应数据继续更新<br>2. 无旧筛选残留数据串入 |
| P1 | 断网后降级行为 | 1. 页面数据更新时断开网络<br>2. 观察 UI 与 WS 状态 | 1. WS 连接断开<br>2. 页面显示离线 / 网络异常提示（若产品有）<br>3. 列表保留缓存数据，不清空为脏态 |
| P1 | 恢复网络后自动重连 | 1. 断网后恢复网络<br>2. 观察 `WS` 新连接和 UI 恢复 | 1. 自动建立新连接或复用现有连接<br>2. 重新发送订阅类消息<br>3. 价格继续更新，无需重进页面 |
| P1 | 后台 / 前台切换后的 WS 恢复 | 1. 页面数据更新时切后台再回前台<br>2. 观察连接状态 | 1. 前台恢复后连接可继续使用或重新建立<br>2. 数据继续推送到当前页面 |
| P1 | 服务端异常消息处理 | 1. 观察是否出现 error frame / 非预期 payload<br>2. 核对页面表现 | 1. 页面不崩溃<br>2. 不出现整块空白或明显错位<br>3. 错误可恢复 |

|--------|------|---------|---------|

## 字段比对建议

优先使用“**同一时刻截取 UI + 最近一条 WS 消息**”的方式比对，不要隔太久。

建议比对规则：

- **价格**：允许仅因 UI 格式化产生的小数位差异
- **涨跌幅**：允许四舍五入差异，但方向必须一致
- **交易额 / 市值 / 流动性**：允许 UI 单位缩写换算差异（K/M/B）
- **更新时间**：如果 UI 有节流，允许 UI 略晚于原始 WS 推送，但不能长期不刷新

---

## 失败归因建议

当发现首页数据异常时，按下面顺序判断：

1. **连接问题**
   - 没建立 WS
   - 握手失败
   - 频繁断连

2. **订阅问题**
   - 没发 subscribe
   - topic 发错
   - 切 tab 后没切订阅

3. **数据解析问题**
   - WS 有数据，但字段解析错
   - symbol 映射错
   - 数值单位 / 精度处理错

4. **UI 映射问题**
   - 收到新值，但没刷到对应行
   - 刷到了错误行
   - 刷新时闪烁 / 整页抖动

5. **容错问题**
   - 断网后无提示
   - 重连后不恢复
   - 缓存态错误清空

## 与自动化结合建议

后续若要脚本化，建议按两段实现：

### 1. 连接与消息采集

- 页面加载前注入 `docs/skills/scripts/inject/websocket-monitor.js`
- 记录：
  - 连接 URL
  - 连接数
  - 最近消息
  - 错误数

### 2. UI 对比断言

- 锁定 1-2 个稳定 symbol
- 从 WS 最近消息提取价格 / 涨跌 / 交易额
- 与页面对应行做实时比对
- 断网 / 恢复网络后再次执行同一断言

---

## 结论口径

首页改用 WebSocket 后，测试通过至少应满足：

- 能看到 WS 地址和消息帧
- 首页关键字段由 WS 推送驱动更新
- UI 与 WS 数据一致
- 断开后能降级，恢复后能重连
- 高频更新下页面仍可用
