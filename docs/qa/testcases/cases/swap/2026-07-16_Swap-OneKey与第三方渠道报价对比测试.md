# Swap - OneKey 询价接口与第三方渠道 API 报价对比测试

> 生成时间：2026-07-16
> 规则文档：`docs/qa/rules/swap-rules.md`、`docs/qa/rules/swap-network-features.md`
> 需求文档：依据 `docs/qa/rules/swap-rules.md` 中报价测试、报价一致性、路由展示与 Provider 规则编写
> 测试端：Web / iOS / Android / Desktop / Extension
> 参考文档：`docs/qa/testcases/cases/swap/2026-07-16_Swap-历史记录测试.md`、`docs/qa/testcases/cases/swap/2026-07-16_Swap-代币税检测测试.md`

## 测试范围说明

**功能范围**：OneKey `Swap` 询价接口返回结果与第三方渠道 API 原始报价的对比验证，包括 `Est Received`、`Rate`、`Network Fee`、`Protocol Fee`、`Provider`、最优报价排序、刷新与失效处理。

**覆盖重点**：
- 同一参数下，OneKey 询价接口返回报价与第三方渠道 API 原始报价结果差距默认控制在 `5%` 以内。
- 多渠道同时返回时，OneKey 默认展示最优报价，不误选次优报价。
- 报价刷新、过期、弱网、单渠道异常等情况下，OneKey 聚合结果不出现脏数据、错排序、错 Provider。
- 同链与跨链均覆盖；对依赖 `quoteResultCtx` 的渠道，同时验证 OneKey 询价接口返回的这一轮报价与后续构建使用的是同一轮有效报价。
- 若价差超过 `5%`，优先定位是否为第三方接口升级、OneKey 未同步升级、费用口径不一致、参数不一致或样本已过期。

## 前置条件

1. 已登录可用钱包；OneKey `Swap` 询价接口、测试网络、第三方渠道 API 服务均可用。
2. 已准备稳定可复现的测试币对与金额样本，至少覆盖：
   - 同链：主币→代币、代币→主币、代币→代币
   - 跨链：主币→主币、主币→代币、代币→主币、代币→代币
3. 对比时必须保持以下参数完全一致：
   - 源链 / 目标链
   - 源币 / 目标币
   - 输入金额
   - 滑点设置
   - 接收地址
   - 钱包类型 / 账户类型
4. 已准备可获取双方报价结果的对照方式，至少满足以下要求：
   - 可直接查看 OneKey 询价接口返回结果，如 `/swap/v1/quote`、`/swap/v1/quote/events`、客户端抓包或调试日志
   - 可直接查看第三方渠道 API 原始报价，如第三方官方 API、返回日志或服务端调试日志
5. 对比时控制在同一时间窗内完成，优先要求在同一轮报价有效期内取值，避免市场波动引入误差。
6. 报价对比默认验收口径：
   - `Est Received`、`Rate` 或按产品定义折算后的最终报价差距应控制在 `5%` 以内
   - 超过 `5%` 视为异常样本，需继续判断是接口升级 / 口径漂移 / 参数不一致 / 市场快速波动中的哪一类原因

## 1. 单渠道报价一致性

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|---|---|---|---|
| ❗️❗️P0❗️❗️ | 1. 当前仅命中 1 个第三方渠道<br>2. 已准备该渠道 API 原始报价结果 | 1. 调用或抓取 OneKey 询价接口返回结果<br>2. 同步获取第三方渠道 API 原始报价 | 1. OneKey 返回的 `Provider` 与第三方渠道名称一致<br>2. `Est Received` 与第三方 API 报价差距在 `5%` 以内<br>3. `Rate` 方向与数值口径一致，差距在 `5%` 以内 |
| ❗️❗️P0❗️❗️ | 1. 当前为同链报价<br>2. 已准备第三方原始同链 API 报价 | 1. 对比 OneKey 询价接口与第三方 API 的 `Est Received`、`Network Fee`、`Rate` | 1. 目标币数量精度符合 decimals 规范<br>2. `Est Received`、`Rate` 差距在 `5%` 以内<br>3. `Network Fee` 主币单位与法币值口径一致<br>4. 不出现 OneKey 比第三方多扣或少展示费用项的情况 |
| ❗️❗️P0❗️❗️ | 1. 当前为跨链报价<br>2. 已准备第三方原始跨链 API 报价 | 1. 对比 OneKey 询价接口与第三方 API 的 `Est Received`、`Network Fee`、`Protocol Fee`、`Est. Time` | 1. `Est Received` 或按费用折算后的最终报价差距在 `5%` 以内<br>2. `Protocol Fee` 与法币折算口径一致<br>3. `Est. Time` 转换后的可读时间与第三方原始时间范围一致<br>4. 不遗漏跨链桥费用或桥名称 |
| ❗️❗️P0❗️❗️ | 1. 当前 OneKey 询价接口与第三方 API 报价差距超过 `5%` | 1. 记录双方报价字段、请求参数、返回时间与版本信息<br>2. 对照近期渠道接口变更或抓包结果 | 1. 该样本标记为异常样本<br>2. 优先排查第三方接口升级、OneKey 未同步升级、费用口径差异或参数不一致<br>3. 不将该结果直接判定为正常市场波动 |
| P1 | 1. 当前渠道为中心化换汇类渠道，如 `HiFi / ChangeHero / Changelly / Swft` 之一 | 1. 输入固定币对与金额完成询价<br>2. 对比第三方原始报价 | 1. OneKey 展示的报价与第三方换汇结果一致<br>2. 页面不误展示 DEX 路由拆分或授权提示 |
| P1 | 1. 当前渠道为 DEX 聚合类渠道，如 `0x / 1inch / OKX / LiFi` 之一 | 1. 输入固定币对与金额完成询价<br>2. 对比第三方原始报价 | 1. OneKey 展示的聚合渠道名称、路由、费用项与第三方结果一致<br>2. 不误展示为其他渠道文案 |

## 2. 多渠道最优报价排序

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|---|---|---|---|
| ❗️❗️P0❗️❗️ | 1. 同一币对可同时返回 2 个及以上第三方渠道报价 | 1. 调用或抓取 OneKey 询价接口返回结果<br>2. 记录所有返回渠道的 `Est Received` | 1. OneKey 默认高亮或默认选中的渠道为 `Est Received` 最大的报价<br>2. 默认最优渠道与对应第三方 API 原始最优报价差距在 `5%` 以内<br>3. 不误选次优报价 |
| ❗️❗️P0❗️❗️ | 1. 多渠道报价存在接近值<br>2. 已获取各渠道 API 原始结果 | 1. 对比 OneKey 返回的渠道排序与各渠道实际报价 | 1. 渠道列表排序与报价高低一致<br>2. 若产品按最优接收量排序，则排序逻辑稳定且可解释 |
| P1 | 1. 多渠道中部分返回 `Provider error` 或无有效报价 | 1. 完成询价<br>2. 观察渠道列表与默认结果 | 1. 无效渠道不应成为默认选中项<br>2. 错误渠道不污染最优报价排序 |
| P1 | 1. 多渠道同时返回<br>2. 存在不同费用结构 | 1. 对比 OneKey 默认选中的最优渠道与第三方原始结果 | 1. OneKey 最终默认结果与“扣除已展示费用后的最优结果”一致<br>2. 不只按单一裸 `toAmount` 误排 |

## 4. 报价刷新与失效处理

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|---|---|---|---|
| ❗️❗️P0❗️❗️ | 1. 当前报价存在有效期与倒计时 | 1. 调用 OneKey 询价接口并记录返回时间<br>2. 观察倒计时至接近过期<br>3. 记录刷新前后报价 | 1. 报价有效期、倒计时展示正确<br>2. 刷新后 `Est Received`、`Rate`、费用字段更新及时<br>3. 过期前后不展示脏数据 |
| ❗️❗️P0❗️❗️ | 1. 当前报价已过期 | 1. 点击 `Swap` 或手动刷新 | 1. 触发重新询价或报价失效提示<br>2. 页面不使用过期报价继续提交 |
| P1 | 1. 市场轻微波动<br>2. 同一币对连续两次询价 | 1. 在短时间内连续触发两次询价 | 1. OneKey 与第三方各自的报价变化方向一致<br>2. 价格波动在可接受范围内，不出现异常跳变<br>3. 若仍在同一时间窗内，对比价差原则上保持在 `5%` 以内 |
| P1 | 1. 已切换金额、滑点或接收地址 | 1. 修改任一参数后重新询价 | 1. 报价按新参数完全刷新<br>2. 不沿用旧参数对应的第三方报价结果 |

## 5. 参数一致性与对比口径

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|---|---|---|---|
| ❗️❗️P0❗️❗️ | 1. 已准备 OneKey 与第三方对比样本 | 1. 分别核对 OneKey 询价接口请求参数与第三方 API 请求参数：链、币对、金额、滑点、接收地址 | 1. 参数完全一致时才进行报价比对<br>2. 任一关键参数不一致时，应记录为无效对比样本<br>3. 参数不一致导致的价差不纳入 `5%` 阈值判断 |
| ❗️❗️P0❗️❗️ | 1. 当前为 ERC20 / SPL / TRC20 / 跨链桥等不同类型资产 | 1. 对比 OneKey 与第三方的精度与单位展示 | 1. 各链主币、代币、法币换算单位一致<br>2. 不因 decimals 不同导致对比误差被误判为渠道差异 |
| P1 | 1. 当前渠道依赖 `quoteResultCtx` | 1. 询价后进入确认页<br>2. 对照当前提交使用的报价轮次 | 1. 页面提交使用的为当前这轮有效报价<br>2. 不出现页面展示 A 轮报价、实际构建使用 B 轮 `quoteResultCtx` 的情况 |

## 6. 异常与降级

| 优先级 | 场景 | 操作步骤 | 预期结果 |
|---|---|---|---|
| ❗️❗️P0❗️❗️ | 1. 某第三方渠道超时或返回错误<br>2. 其他渠道仍可正常报价 | 1. 调用 OneKey 询价接口 | 1. OneKey 聚合报价仍可返回有效结果<br>2. 异常渠道不成为默认渠道<br>3. 返回结果中不出现脏的 `Est Received` 或错 Provider<br>4. 不因单渠道异常把 OneKey 与有效第三方渠道的正常价差误判为超 `5%` 问题 |
| P1 | 1. 所有第三方渠道均无有效报价 | 1. 完成询价 | 1. 页面展示无报价或错误提示<br>2. 不保留上一笔报价结果 |
| P1 | 1. 某第三方渠道返回有效 `Provider` 标识但无有效金额 | 1. 完成询价 | 1. 该渠道不视为有效报价<br>2. 不参与默认排序与结果展示 |
| P1 | 1. 弱网或断网环境 | 1. 触发询价并观察恢复过程 | 1. 页面展示错误提示、重试入口或 loading 兜底<br>2. 网络恢复后可重新获取有效报价 |
