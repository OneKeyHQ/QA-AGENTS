> 模块：Account / Hardware
> 来源：Trezor 连接需求说明 + `app-monorepo` `x` 分支实现对齐
> App 版本：待定
> 日期：2026-07-13
> 测试端：Desktop（USB / BLE，按设备能力生效）

---

## 1. 需求背景

Trezor 接入需要统一多设备发现、USB/BLE 双通道绑定、设备初始化校验、PIN/安全码、Passphrase 隐藏钱包、地址派生上限和设备管理行为。Trezor 不使用 Ledger 式链 App 安装模型，连接后可直接按支持网络发起地址派生和签名交互。

本需求沉淀产品需求与业务边界；需求与规则已确认，对应用例见 `docs/qa/testcases/cases/hardware/2026-07-14_Hardware-第三方硬件-Trezor连接与设备交互.md`。

---

## 2. 功能范围与基础口径

| 项目 | 说明 |
|-----|------|
| 支持网络 | BTC、EVM（ETH）、TRON（TRX）、Solana（SOL） |
| 支持连接 | Desktop 支持 USB；具备 BLE 能力的设备支持蓝牙连接 |
| 多设备 | 允许发现和连接多台 Trezor，不沿用 Ledger USB「仅允许一台」限制 |
| 设备身份 | 以设备固件返回的 `device_id`（下文称 DeviceID）作为钱包与设备匹配的核心身份；USB/BLE connectId 仅表示连接通道 |
| App 安装 | 不适用；不展示 Ledger 式链 App 安装、批量安装或单网络补装流程 |
| 客户端并发 | 同一台 Trezor 同一时刻只允许一个客户端持有交互会话；被其他客户端占用时，本客户端需提示设备忙并中断当前操作 |

## 3. 连接、识别与钱包匹配

### 3.1 多设备连接

1. 当前连接方式页面允许展示多台同通道的 Trezor，并保留设备名称、型号等可区分信息。
2. 用户选择某台设备后，只对所选设备建立当前交互会话；其他已发现设备不应阻塞连接。
3. USB 与蓝牙分属两个独立连接页面；USB 页面只搜索和展示 USB 设备，蓝牙页面只搜索和展示 BLE 设备，不同时发现或混合展示两类结果。
4. 同一台设备分别从 USB 页面和蓝牙页面连接时，仍应根据 DeviceID 识别为同一设备，不应创建为两台独立设备。

### 3.2 重复添加与设备重置

| 场景 | 结果 |
|-----|------|
| 同一台未重置设备重复添加标准钱包 | 根据相同 DeviceID 定位并复用已添加钱包，刷新连接元数据，不创建重复标准钱包 |
| 同一台设备重置后重新添加 | 重置后 DeviceID 变化，创建新钱包 |
| 重置后恢复相同助记词 | 仍按新 DeviceID 创建新钱包，不因助记词相同覆盖旧钱包 |
| 另一台设备使用相同助记词 | 视为另一设备，不与原设备钱包合并 |

### 3.3 USB 钱包绑定 BLE

1. 已通过 USB 创建的钱包可在设备支持 BLE 且 Desktop BLE 可用时发起蓝牙绑定。
2. App 扫描 BLE 候选设备后，需要连接候选设备并读取 DeviceID。
3. 只有候选设备 DeviceID 与 USB 创建钱包记录的 DeviceID 一致时，才允许把 BLE connectId 写入同一设备记录。
4. 以下场景均不得绑定：
   - 原设备重置后，DeviceID 已变化；
   - 另一台设备导入相同助记词；
   - 选择了其他 Trezor 设备；
   - 候选设备要求建立新的配对关系，无法证明是原设备。
5. 已保存的 BLE 标识失效时，用户仍可重新进入绑定流程并覆盖旧 BLE connectId。

## 4. Onboarding、PIN 与安全码

### 4.1 未初始化设备

连接后若设备返回 `initialized = false`：

1. App 显示 Trezor 尚未初始化的提示；
2. 中断钱包创建，不生成临时或正式钱包；
3. 引导用户先在 Trezor 官方流程完成设备初始化，再返回 OneKey 重试；
4. App 不代替 Trezor 执行第三方设备初始化。

### 4.2 旧设备 PIN

Trezor Model One 等使用按键和乱序矩阵的旧设备，需要在 App 输入 PIN 位置序列；App 不展示设备上的实际数字映射，提交结果以设备校验为准。PIN 错误、取消或设备断开时，中断当前交互并显示对应错误。

### 4.3 Safe 7 首次安全码校验

Trezor Safe 7 与 App 首次建立 THP 配对时：

1. 设备显示安全码；
2. App 展示安全码输入框，用户输入设备上显示的安全码后提交；
3. 空值不能提交；错误安全码、取消或超时不得建立配对；
4. 配对成功后保存该设备的配对凭证，后续连接可复用；移除设备或设备重置后需重新配对。

## 5. Passphrase 与隐藏钱包

1. Trezor 支持使用 Passphrase 创建隐藏钱包；空 Passphrase 等价于标准钱包。
2. 每个隐藏钱包必须绑定对应的 `passphraseState`，派生地址、签名和钱包指纹计算均使用相同状态。
3. 相同设备输入相同 Passphrase 时定位到同一隐藏钱包，不创建地址不同的重复钱包。
4. Safe 7 支持在有效会话内保持多个隐藏钱包状态，切换已保持的钱包时无需重新输入 Passphrase。
5. 早期设备同一时间只保持一个隐藏钱包交互状态；切换到其他隐藏钱包时需重新输入 Passphrase。
6. `passphraseState` 失效或不匹配时必须要求重新输入，不得使用错误状态继续派生地址或签名。

## 6. 地址派生序号限制

| 网络 / 派生模式 | 当前需求给出的限制阈值 |
|----------------|----------------------|
| ETH BIP44 | 1,000,000 |
| TRON | 1,000,000 |
| SOL | 100 |
| ETH Ledger Live 路径 | 100 |
| ETH Ledger Legacy 路径 | 100 |
| BTC | 当前不设业务限制 |

说明：

- App 在超过设备/SDK允许范围时应终止派生，并显示可理解的路径或序号不支持提示，不得停留在永久 loading。
- 当前输入资料只给出限制阈值，尚未明确阈值本身是否可用；正式编写用例前需确认 `阈值 - 1 / 阈值 / 阈值 + 1` 的实际边界。
- BTC 虽无当前业务限制，仍受 BIP32 序号格式、设备性能和 SDK 参数合法性约束。

## 7. 设备管理

1. 已连接的 Trezor 可进入设备详情，展示设备名称、型号、固件版本和连接状态等可用信息。
2. 设备初始化时未手动命名、设备 label 为空时，App 使用设备型号作为设备展示名称，不显示空名称。
3. 同一 DeviceID 的设备名称在第二个 OneKey App 或 Trezor Suite 中被修改后，当前 App 在下一次读取设备信息的交互中（例如核对地址）同步最新设备名称；与该设备关联的钱包名称和设备详情均使用最新名称。
4. 可修改项目由设备型号、固件能力字段和 SDK 能力共同决定；不支持的设置不展示，或明确显示不可用。
5. 可用能力可能包括自动锁定、亮度、触觉反馈、PIN 和 Passphrase 等；不同设备不要求展示完全一致的设置集合。
6. 设置操作需由目标设备确认；设备拒绝、断开或方法不支持时，不更新 App 本地展示状态。
7. Trezor 不展示 OneKey 专属设置、固件验真或 Ledger App 管理入口。

## 8. 异常与恢复

| 异常 | 处理要求 |
|-----|---------|
| 设备被其他客户端占用 | 提示设备忙/正被其他应用使用，终止当前操作；其他客户端释放后允许重试 |
| 设备未初始化 | 提示先初始化并中断创建 |
| USB/BLE 断开 | 终止当前交互，保留已完成数据；恢复连接后允许重试 |
| BLE 候选 DeviceID 不匹配 | 不绑定该候选设备，允许继续选择其他候选设备 |
| 安全码错误/取消/超时 | 不保存配对关系，允许重新发起配对 |
| PIN 错误/不匹配 | 显示 PIN 错误语义，不归类为通用连接失败 |
| Passphrase 状态不匹配 | 要求重新输入 Passphrase，不继续派生或签名 |
| 派生序号超限 | 显示路径/序号不支持错误并结束 loading |

## 9. 验收标准（Definition of Done）

1. Desktop 可通过 USB 连接 Trezor；Safe 7 等支持 BLE 的设备可通过蓝牙连接。
2. USB、蓝牙连接页面相互独立；同一页面可发现多台当前通道的 Trezor，选择目标设备后可独立完成连接。
3. 同一未重置设备重复添加时定位到已有钱包；设备重置后创建新钱包。
4. USB 钱包只有在 BLE 候选 DeviceID 一致时绑定成功。
5. 未初始化设备被明确提示并中断创建。
6. Model One App 端 PIN 矩阵和 Safe 7 首次安全码流程按设备能力触发。
7. BTC / EVM / TRON / SOL 地址派生与签名流程可达，超限路径返回明确错误。
8. 标准钱包和 Passphrase 隐藏钱包不会因会话状态串用而派生出错误地址。
9. 未手动命名且设备 label 为空时，App 按设备型号显示设备名称。
10. 设备在第二个 OneKey App 或 Trezor Suite 中改名后，当前 App 在核对地址等设备交互后同步最新设备名称。
11. 设备管理只展示当前型号实际支持的设置能力。
12. 同设备被其他客户端占用时返回设备忙提示，释放后可恢复连接。

## 10. 源码对齐与待确认项

### 10.1 已在 `app-monorepo` `x` 分支对齐的行为

- Trezor UI 提供 USB、蓝牙两个独立连接页面，并支持多设备结果展示。
- 钱包和设备记录使用固件 DeviceID，并区分 USB/BLE connectId。
- USB→BLE 绑定通过 DeviceID 校验，同助记词不能替代设备身份。
- 未初始化设备会弹提示并中断钱包创建。
- 重置后 DeviceID 变化会创建新钱包。
- 设备 label 为空时，第三方硬件展示名称由设备型号生成；Trezor features 事件会按 DeviceID 更新本地设备信息并触发关联钱包名称同步。
- 已存在 Trezor PIN 矩阵、安全码输入、Passphrase 状态、设备忙错误和按能力展示设备设置的实现结构。
- BTC / EVM / TRON / SOL 均存在 Trezor keyring 适配。

### 10.2 待确认或待实现差异

| 项目 | 当前状态 |
|-----|---------|
| USB/BLE 搜索实现 | 产品规则要求各页面只搜索当前通道；`x` 分支 `ConnectionFlowTrezor` 当前仍有共享扫描后按页面过滤结果的实现，需要开发对齐 |
| 旧设备范围 | `x` 分支厂商配置注释仍以 Safe 7 为当前主支持对象；Model One 的正式启用范围需产品/开发确认 |
| 多隐藏钱包会话机型差异 | 已有 `passphraseState` 传递与复用结构，但 Safe 7 多会话、旧设备单会话的完整机型矩阵需联调确认 |
| 派生阈值包含关系 | 1,000,000 与 100 是否为可用最大值尚未明确 |
| 设备设置矩阵 | 需按各型号实际 features/SDK 能力确认最终展示项 |
