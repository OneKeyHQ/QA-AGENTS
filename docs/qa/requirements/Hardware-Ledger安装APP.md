## 1. 需求背景

Ledger 当前已具备设备识别、账户匹配与四链交互能力（BTC / EVM / SOL / TRON）。本次新增能力是在已有链路上补充“Ledger App 安装”流程，使用户在设备缺少目标网络 App 时，可以在创建账户或单网络交互过程中完成按需安装，并在异常情况下得到明确提示。

本次需求目标是：
- 保持现有“设备识别 / 设备不匹配 / BLE 聚合”规则不回退。
- 在 Onboarding 与网络选择投资组合场景下，支持基于设备当前已安装 App 状态决定是否进入批量安装流程。
- 在单网络交互场景下，支持按网络提示并安装对应 App，再继续原业务流程。
- 对断网、设备空间不足、设备断线等安装失败场景给出明确提示，并可感知失败原因。

## 2. 功能范围与基础口径

### 2.1 支持网络范围

| 项目 | 说明 |
|-----|------|
| 当前安装能力覆盖网络 | BTC / ETH / TRX / SOL |
| 投资组合勾选网络 | 仅上述四个网络参与 Ledger App 安装判断 |
| 其他网络 | 不进入本次 Ledger App 安装能力范围 |

### 2.2 名词口径

| 名词 | 说明 |
|-----|------|
| 可用 App | 设备上已安装，且能支撑当前网络后续创建地址 / 核对地址 / 签名交互的 Ledger App |
| 安装流程 | App 侧提示用户安装目标网络对应 Ledger App，并在设备侧按顺序完成安装 |
| 单网络交互 | 创建地址、核对地址、签名等只针对单一网络发起的硬件交互 |
| 投资组合创建地址 | 在投资组合页勾选多个网络后批量创建地址；实际执行时按连续单网络交互处理 |
| 网络命名口径 | 安装能力按 App 名称使用 `ETH / TRX`；与既有账户交互文档中的 `EVM / TRON` 属同一能力域的不同描述口径 |

### 2.3 适用场景

| 场景 | 说明 |
|-----|------|
| Onboarding 创建账户 | 首次接入 Ledger 时，在流程中创建账户 |
| 网络选择投资组合创建账户 | 旧 all networks 场景，对所勾选网络批量创建地址 |
| 单网络创建地址 | 在某一网络下单独发起创建地址 |
| 单网络核对地址 | 在某一网络下发起地址校验 |
| 单网络签名 | 在某一网络下发起签名或交易确认 |

## 3. Ledger App 安装流程

### 3.1 场景一：Onboarding / 投资组合创建账户

#### 3.1.1 触发前提

| 条件 | 是否进入安装流程 |
|-----|----------------|
| BTC / ETH / TRX / SOL 四个网络的 App 全部未安装 | 进入“复选框选择需要安装的 App”流程 |
| 四个网络中至少存在 1 个可用 App | 不进入批量安装流程 |

#### 3.1.2 批量安装入口规则

| 项目 | 说明 |
|-----|------|
| 弹窗前提 | 仅当 4 个支持网络 App 全部未安装时弹出 |
| 选择方式 | 以复选框形式选择需要安装的 App |
| 默认目的 | 用户选择后按顺序安装所选 App |
| 安装顺序 | 对已勾选的网络按产品定义顺序逐个安装，前一个完成后再进入下一个 |

#### 3.1.3 存在可用 App 时的处理

| 场景 | 规则 |
|-----|------|
| 至少存在 1 个可用 App | 不显示批量安装复选框 |
| 投资组合勾选多个网络 | 仅对已安装 App 对应网络继续创建 |
| 未安装 App 的网络 | 不创建地址，不应伪成功 |

### 3.2 场景二：单网络交互按需安装

#### 3.2.1 适用范围

| 交互类型 | 规则 |
|---------|------|
| 创建地址 | 目标网络 App 未安装时，提示安装对应 App |
| 核对地址 | 目标网络 App 未安装时，提示安装对应 App |
| 签名 | 目标网络 App 未安装时，提示安装对应 App |

#### 3.2.2 单网络安装规则

| 场景 | 规则 |
|-----|------|
| 当前交互网络 App 已安装 | 直接进入原交互流程 |
| 当前交互网络 App 未安装 | 提示安装对应网络 App |
| 安装完成 | 返回并继续当前单网络交互 |
| 安装取消 / 失败 | 停留在当前业务上下文，并给出失败提示 |

#### 3.2.3 投资组合创建地址按单网络交互处理

| 项目 | 说明 |
|-----|------|
| 执行模型 | 投资组合勾选网络创建地址，按连续单网络交互处理 |
| 统一结果页 | 不采用“一个结果页统一处理所有未安装网络”的模式 |
| 网络处理顺序 | 对每个勾选网络，按单网络交互顺序逐个处理是否需要安装 App |
| 已安装网络 | 逐个进入对应网络的创建地址流程 |
| 未安装且未补装成功网络 | 不创建地址，不纳入成功创建结果 |
| 前序成功网络 | 不因后续网络未安装或安装失败而回滚 |

## 4. 异常场景与提示口径

### 4.1 异常类型

| 异常类型 | 触发时机 | 期望口径 |
|---------|---------|---------|
| 断网 | 安装前请求、安装中请求或安装完成回传依赖网络时发生 | 提示网络异常 / 安装失败，并允许用户感知失败原因 |
| 设备空间不足 | 设备侧无法继续安装目标 App | 提示设备空间不足，安装失败 |
| 设备断线 | 安装过程中拔线、蓝牙断开、设备掉线 | 提示设备已断开或连接中断，安装失败 |

### 4.2 异常处理原则

| 项目 | 说明 |
|-----|------|
| 提示要求 | 必须区分异常原因，不可统一为模糊失败提示 |
| 成功结果保护 | 已成功安装 / 已成功创建的网络结果保持有效，不因后续网络失败而回滚 |
| 失败网络处理 | 当前失败网络停止继续创建地址 / 核对地址 / 签名 |
| 用户感知 | 用户能够明确知道失败原因来自网络、设备空间或设备连接 |

## 5. 回归范围

| 类别 | 回归重点 |
|-----|---------|
| 安装流程 | 批量安装入口、单网络按需安装、安装顺序、安装后继续原流程 |
| 四链交互 | BTC / ETH / TRX / SOL 下，创建地址、核对地址、签名 / 转账流程可达 |
| 异常提示 | 断网、空间不足、设备断线三类提示准确且时机正确 |
| 兼容性 | 不影响现有设备识别、设备不匹配、BLE 聚合 / 新建分支 |

## 6. 验收标准（Definition of Done）

1. 仅在 BTC / ETH / TRX / SOL 四个支持网络 App 全部未安装时，Onboarding / 投资组合场景出现批量安装复选框。  
2. 只要四个支持网络中存在至少一个可用 App，就不进入批量安装流程，并仅按已安装 App 对应网络创建地址。  
3. 单网络创建地址 / 核对地址 / 签名场景下，目标网络 App 未安装时，能够提示安装对应 App，并在安装完成后继续原交互。  
4. 投资组合勾选多个网络时，整体执行按连续单网络交互处理；某网络未安装或安装失败时，仅该网络不创建，不影响其他已成功网络。  
5. 断网、设备空间不足、设备断线三类异常均有对应提示，且用户可区分失败原因。  
6. 新增安装能力不影响既有设备识别、设备不匹配、BLE 聚合 / 新建 wallet 规则。
