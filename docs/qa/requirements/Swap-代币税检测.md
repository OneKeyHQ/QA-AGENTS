# Swap - 代币税检测需求文档

> 模块：Swap
> 功能：代币税检测 / 风险提示 / 默认滑点自动调整
> 版本：
> 测试端：Web / iOS / Android / Desktop / Extension

---

## 需求背景

Trade 当前的代币税检测依赖 GoPlus 单一数据源，存在漏检。漏检时交易会沿用用户常规滑点配置（如 1%）发起，实际到账金额因代币税低于最小到账量，链上交易大概率回退，用户难以从界面上理解失败原因。

本需求在现有 GoPlus 检测之外，引入 OKX 作为第二层代币税检测数据源，降低漏检率；当检测到代币税时，页面需向用户展示税率提示，并自动上调默认滑点，减少因税费导致的失败。

---

## 功能描述

### 1. 双数据源代币税检测
- 现有 GoPlus 检测逻辑保留。
- 新增 OKX 报价接口作为第二层代币税检测来源。
- 任一数据源、任一方向税率大于 0，即视为命中代币税检测。
- 当两个数据源均返回税率时，取较大值作为生效税率。
- OKX 失败、超时或无数据时，静默降级为仅使用 GoPlus 结果，不阻塞报价，不额外展示错误。

### 2. 风险提示展示
- 检测到代币税时，在交易页底部展示提示信息。
- 提示文案包含检测到的税率数值。
- 参考文案：`检测到该代币存在 {amount}% 代币税，滑点已自动上调以确保交易成功`
- 未检测到代币税时，不新增任何提示。

### 3. 默认滑点自动调整
- 当用户未手动修改滑点时，当前生效滑点 = 用户原有滑点设置 + 生效税率。
- 示例：原滑点 1%，生效税率 5%，则当前生效滑点显示为 6%。
- 滑点设置入口展示当前生效值，用户可查看、可修改。

### 4. 用户手动滑点优先
- 用户手动修改滑点后，以用户输入值为准。
- 检测结果晚于用户手动输入返回时，不得覆盖用户当前输入值。
- 自动叠加逻辑仅作用于默认值，不作用于用户已编辑值。

### 5. 重新检测与重算
- 切换代币后，需要重新发起代币税检测。
- 切换交易方向后，需要重新发起代币税检测。
- 检测结果更新后，若用户未手动修改滑点，则按最新结果重新计算默认滑点。

### 6. 特殊场景
- GoPlus 与 OKX 返回不同税率时，以较高税率作为前端展示和默认滑点叠加依据。
- from / to 两侧都存在代币税时，页面进入代币税提示态；展示税率与默认滑点叠加值以前后端约定的生效税率字段为准，前端不自行推导额外规则。
- 高税率代币（如 30%+）需继续展示提示与当前生效滑点；是否设置自动调整上限、是否增加二次确认，当前版本未定义。
- 条件触发型代币税（如仅特定池路径收税）仍可能被双源同时漏检，本需求目标为降低漏检率，不承诺完全覆盖。

---

## 业务规则

| 规则 | 描述 |
|------|------|
| 检测命中 | 任一数据源、任一方向税率大于 0 即命中 |
| 生效税率 | 多数据源同时返回时取较大值；from / to 双向有税时以前后端约定的生效税率字段为准 |
| OKX 异常 | 失败 / 超时 / 无数据时静默降级，不阻塞报价 |
| 风险提示 | 仅命中代币税时展示；未命中不展示 |
| 默认滑点 | 用户未手动修改时，默认滑点 = 原滑点设置 + 生效税率 |
| 手动优先 | 用户手动修改后，不再被检测结果覆盖 |
| 重算时机 | 切代币、切方向后重新检测并重算默认滑点 |

---

## 已知风险

- 双源均可能漏检条件触发型代币税，无法替代真实路由级模拟。
- 高税率代币可能带来异常大的默认滑点，当前版本未定义上限与二次确认策略。
- 两源税率不一致时，取较高值可能使当前生效滑点高于实际所需值，用户实收可能优于预估。
